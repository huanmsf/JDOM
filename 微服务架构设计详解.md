# Part 1: 微服务架构概述

## 1.1 什么是微服务

微服务（Microservices）是一种软件架构风格，将应用程序拆分为一组小型、独立部署、通过轻量级通信协议（如 HTTP/gRPC）相互通信的服务。每个服务运行在自己的进程中，围绕业务能力构建，通过自动化部署机制独立部署。

```
+------------------------------------------------------------------+
|              单体架构 vs 微服务架构                              |
+------------------------------------------------------------------+
|                                                                  |
|  单体架构:                    微服务架构:                        |
|  +-------------------+        +---------+ +---------+           |
|  |   单体应用        |        | 用户服务 | | 订单服务 |          |
|  | +---+ +---+ +---+ |        +---------+ +---------+           |
|  | |UI | |BL | |DB | |        +---------+ +---------+           |
|  | +---+ +---+ +---+ |        | 商品服务 | | 支付服务 |          |
|  +-------------------+        +---------+ +---------+           |
|         单个部署                 各自独立部署                     |
+------------------------------------------------------------------+
```

## 1.2 微服务核心特征

| 特征 | 描述 |
|------|------|
| 单一职责 | 每个服务只负责一个业务能力 |
| 独立部署 | 每个服务可以独立构建、测试、发布 |
| 去中心化 | 数据管理去中心化，每个服务拥有独立数据库 |
| 故障隔离 | 一个服务的故障不影响其他服务 |
| 技术异构 | 不同服务可以使用不同技术栈 |
| 自动化 | CI/CD 管道自动化测试和部署 |
| 轻量通信 | 服务间通过 REST/gRPC/消息队列通信 |

## 1.3 微服务 vs 单体 vs SOA

```
+--------------------+----------+----------+----------+
| 维度               | 单体架构 | SOA      | 微服务    |
+--------------------+----------+----------+----------+
| 服务粒度           | 粗       | 粗       | 细        |
| 通信协议           | 内部调用 | ESB/SOAP | REST/gRPC |
| 数据管理           | 共享DB   | 共享DB   | 独立DB    |
| 部署单元           | 整体     | 服务模块 | 每个服务  |
| 开发团队           | 大团队   | 中团队   | 小团队    |
| 部署频率           | 低       | 低       | 高（日）  |
| 技术栈             | 统一     | 统一     | 多样      |
| 运维复杂度         | 低       | 中       | 高        |
+--------------------+----------+----------+----------+
```

## 1.4 微服务的挑战

```
微服务带来的新挑战:

  1. 分布式系统复杂性
     - 网络分区、延迟、消息丢失
     - 分布式事务（两段提交、Saga）
     - 数据一致性（最终一致性 vs 强一致性）

  2. 运维复杂性
     - 服务发现与负载均衡
     - 分布式链路追踪
     - 多服务日志聚合
     - 容器编排（Kubernetes）

  3. 测试复杂性
     - 契约测试（Consumer-Driven Contract）
     - 集成测试环境难以搭建
     - 服务间依赖复杂

  4. 团队组织挑战
     - 康威定律（Conway's Law）
     - 服务边界难以划定
     - 技术标准统一
```

## 1.5 何时采用微服务

```
适合微服务的场景:
  - 团队规模 > 50 人，需要并行开发
  - 不同模块有差异化的扩展需求
  - 需要多语言/技术栈支持
  - 已有成熟的 DevOps 文化和 CI/CD 管道
  - 业务复杂度高，单体维护成本高

不适合微服务的场景:
  - 初创团队/MVP 阶段（过度设计）
  - 团队规模小（< 10 人）
  - 没有 DevOps 经验和 Kubernetes 运维能力
  - 业务逻辑简单，不需要独立扩展
  - 数据强一致性要求高（分布式事务复杂）
```

# Part 2: 服务拆分原则与策略

## 2.1 服务拆分的核心原则

### 2.1.1 单一职责原则（SRP）

每个服务应该只有一个"变更原因"。如果一个服务因多个不相关的原因而修改，则需要进一步拆分。

```
错误的拆分（职责混乱）:
  用户服务 = 用户注册/登录 + 购物车 + 收货地址 + 用户积分

正确的拆分（单一职责）:
  用户服务    = 用户注册/登录/信息管理
  购物车服务  = 购物车增删改查
  地址服务    = 收货地址管理
  积分服务    = 积分获取/消费/明细
```

### 2.1.2 高内聚低耦合

```
高内聚: 相关联的功能放在同一个服务中
  - 用户的注册、登录、密码重置应在同一服务
  - 避免"数据碎片"：一个业务实体的属性分散在多个服务

低耦合: 服务间依赖最小化
  - 避免服务间直接共享数据库
  - 通过事件驱动替代直接 RPC 调用（异步解耦）
  - 接口稳定，变更时向后兼容
```

### 2.1.3 Database Per Service 原则

```
每个微服务拥有独立数据库（逻辑隔离或物理隔离）:

+------------+     +------------+     +------------+
|  用户服务  |     |  订单服务  |     |  商品服务  |
+-----+------+     +-----+------+     +-----+------+
      |                  |                   |
+-----v------+     +-----v------+     +------v-----+
|  user_db   |     | order_db   |     | product_db |
+------------+     +------------+     +------------+

好处:
  - 独立演化：可以独立选择数据库类型（MySQL/MongoDB/Redis）
  - 防止数据层耦合：不能通过 JOIN 查询其他服务的数据
  - 独立扩展：可以对数据库进行独立的分库分表

代价:
  - 不能使用跨库 JOIN
  - 需要通过 API 组合查询分散在多个服务的数据
  - 分布式事务更复杂
```

## 2.2 服务拆分的策略

### 2.2.1 按业务能力拆分

```
电商系统按业务能力拆分示例:

  用户管理能力   --> 用户服务
  商品管理能力   --> 商品服务
  库存管理能力   --> 库存服务
  订单管理能力   --> 订单服务
  支付能力       --> 支付服务
  物流跟踪能力   --> 物流服务
  消息通知能力   --> 通知服务
  搜索能力       --> 搜索服务
  推荐能力       --> 推荐服务
```

### 2.2.2 按 DDD 限界上下文拆分

```
限界上下文（Bounded Context）是 DDD 中最重要的战略模式，
每个限界上下文对应一个微服务。

  销售上下文     --> 订单服务 + 购物车服务
  目录上下文     --> 商品服务 + 分类服务
  仓储上下文     --> 库存服务 + 仓库服务
  身份认证上下文 --> 用户服务 + 权限服务
  财务上下文     --> 支付服务 + 账单服务
```

### 2.2.3 按可扩展性需求拆分

```
根据服务的流量特征和扩展需求拆分:

  高并发读取  --> 商品详情服务（可独立横向扩展）
  高并发写入  --> 订单创建服务（可独立扩容）
  CPU 密集    --> 图片处理服务（可配置高 CPU 实例）
  IO 密集     --> 文件存储服务（可配置大内存实例）

  原则：有差异化扩展需求的模块拆分为独立服务
```

## 2.3 服务拆分的粒度控制

```
拆分过细（Nano Services）的问题:
  - 服务间调用链过长，延迟累积
  - 分布式事务复杂度指数级增长
  - 运维成本高（N个服务的部署、监控、告警）
  - 团队边界模糊，服务职责重叠

拆分过粗（Mini-Monolith）的问题:
  - 失去了微服务的独立扩展优势
  - 服务体积庞大，启动慢，CI/CD 慢
  - 不同业务的变更频率不匹配，互相阻塞

合理粒度的判断标准:
  - 2 Pizza Team: 2 个披萨能喂饱维护这个服务的团队（6~10 人）
  - 可在 1 周内完全理解该服务
  - 独立部署不超过 30 分钟
```

## 2.4 单体拆分为微服务的路线

```
单体到微服务的迁移步骤（绞杀者模式 - Strangler Fig）:

  阶段1: 模块化单体
    在单体内部通过包/模块清晰划分边界，禁止跨模块直接访问数据库

  阶段2: 抽取最易拆分的服务
    选择依赖最少、变更频率高的模块先拆（如通知服务、文件服务）

  阶段3: 逐步拆分核心服务
    按业务价值优先级，逐步将核心服务剥离
    新功能在新服务开发，旧功能逐步迁移

  阶段4: 拆分数据库
    先逻辑分库（同一MySQL，不同Schema），再物理分库
    通过 Change Data Capture (CDC) 数据同步

  阶段5: 下线单体
    全部业务迁移完成后，逐步停用单体
```

# Part 3: DDD 领域驱动设计

## 3.1 DDD 核心概念

### 3.1.1 实体（Entity）

有唯一标识符的对象，标识符在对象生命周期内不变。

```java
@Entity
public class Order {
    @Id
    private OrderId id;     // 唯一标识
    private UserId userId;  // 关联用户
    private List<OrderItem> items;
    private OrderStatus status;
    private Money totalAmount;

    // 实体通过 ID 判断相等性
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Order)) return false;
        return id.equals(((Order) o).id);
    }
}
```

### 3.1.2 值对象（Value Object）

没有唯一标识符，通过属性值判断相等性，不可变。

```java
@Embeddable
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("Amount cannot be negative");
        }
        this.amount = amount;
        this.currency = currency;
    }

    public Money add(Money other) {
        if (!currency.equals(other.currency)) {
            throw new IllegalArgumentException("Currency mismatch");
        }
        return new Money(amount.add(other.amount), currency);
    }

    // 值对象通过属性值判断相等性
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Money)) return false;
        Money m = (Money) o;
        return amount.equals(m.amount) && currency.equals(m.currency);
    }
}
```

### 3.1.3 聚合根（Aggregate Root）

聚合是一组相关对象的集合，聚合根是聚合的入口。外部只能通过聚合根访问聚合内的对象。

```java
public class Order {  // Order 是聚合根

    private OrderId id;
    private List<OrderItem> items = new ArrayList<>();  // OrderItem 在聚合内
    private OrderStatus status;
    private List<DomainEvent> domainEvents = new ArrayList<>();

    // 业务方法包含在聚合根中
    public void addItem(ProductId productId, int quantity, Money price) {
        if (status != OrderStatus.PENDING) {
            throw new IllegalStateException("Cannot add items to confirmed order");
        }
        // 检查是否已有该商品
        items.stream()
            .filter(item -> item.getProductId().equals(productId))
            .findFirst()
            .ifPresentOrElse(
                item -> item.increaseQuantity(quantity),
                () -> items.add(new OrderItem(productId, quantity, price)));
    }

    public void confirm() {
        if (items.isEmpty()) {
            throw new IllegalStateException("Order must have at least one item");
        }
        status = OrderStatus.CONFIRMED;
        // 发布领域事件
        domainEvents.add(new OrderConfirmedEvent(id, calculateTotal()));
    }

    public Money calculateTotal() {
        return items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.ZERO, Money::add);
    }
}
```

### 3.1.4 领域服务（Domain Service）

不自然属于任何实体或值对象的业务逻辑。

```java
// 转账操作涉及两个账户，不属于单个账户，适合放在领域服务
@DomainService
public class TransferDomainService {

    public void transfer(Account fromAccount, Account toAccount, Money amount) {
        fromAccount.debit(amount);
        toAccount.credit(amount);
    }
}
```

### 3.1.5 限界上下文（Bounded Context）

```
限界上下文定义了模型的适用边界。同一个概念在不同上下文有不同含义:

  "Product" 在不同上下文的含义:
    目录上下文（Catalog）: 商品名称、描述、图片、分类
    销售上下文（Sales）:   商品价格、促销、库存状态
    仓储上下文（Warehouse）: SKU、重量、尺寸、存放位置
    物流上下文（Shipping）: 包裹信息、体积重

  各上下文对 Product 有自己的模型，通过上下文映射（Context Map）集成
```

## 3.2 领域事件（Domain Events）

```java
// 领域事件基类
@Data
public abstract class DomainEvent {
    private final String eventId = UUID.randomUUID().toString();
    private final LocalDateTime occurredAt = LocalDateTime.now();
    public abstract String getEventType();
}

// 订单确认事件
@Data
@EqualsAndHashCode(callSuper = true)
public class OrderConfirmedEvent extends DomainEvent {
    private final String orderId;
    private final String userId;
    private final BigDecimal totalAmount;

    @Override
    public String getEventType() { return "order.confirmed"; }
}

// 在应用服务中发布事件
@Service
@Transactional
public class OrderApplicationService {

    private final OrderRepository orderRepository;
    private final ApplicationEventPublisher eventPublisher;

    public void confirmOrder(String orderId) {
        Order order = orderRepository.findById(OrderId.of(orderId))
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        order.confirm();
        orderRepository.save(order);

        // 发布领域事件（事务提交后）
        order.getDomainEvents().forEach(eventPublisher::publishEvent);
        order.clearDomainEvents();
    }
}
```

## 3.3 仓储模式（Repository Pattern）

```java
// 领域层接口（不依赖具体技术）
public interface OrderRepository {
    Optional<Order> findById(OrderId id);
    List<Order> findByUserId(UserId userId);
    Order save(Order order);
    void delete(OrderId id);
}

// 基础设施层实现（JPA）
@Repository
public class JpaOrderRepository implements OrderRepository {

    private final OrderJpaRepository jpaRepo;
    private final OrderMapper mapper;

    @Override
    public Optional<Order> findById(OrderId id) {
        return jpaRepo.findById(id.getValue())
            .map(mapper::toDomain);
    }

    @Override
    public Order save(Order order) {
        OrderEntity entity = mapper.toEntity(order);
        OrderEntity saved = jpaRepo.save(entity);
        return mapper.toDomain(saved);
    }
}
```

# Part 4: 服务通信

## 4.1 同步通信 vs 异步通信

```
+------------------+-------------------+--------------------+
| 维度             | 同步（REST/gRPC）  | 异步（消息队列）    |
+------------------+-------------------+--------------------+
| 耦合程度         | 时间耦合           | 解耦               |
| 响应时间         | 实时               | 延迟不确定         |
| 错误处理         | 简单（异常）       | 复杂（死信队列）   |
| 适用场景         | 需要立即结果       | 可接受最终一致     |
| 服务可用性       | 强依赖             | 弱依赖             |
+------------------+-------------------+--------------------+

选择原则:
  - 查询操作 → REST/gRPC
  - 需要即时反馈的命令 → REST/gRPC（如创建订单返回订单号）
  - 跨服务状态更新 → 消息队列（如订单确认后通知库存扣减）
  - 高并发写入 → 消息队列削峰填谷
```

## 4.2 OpenFeign 远程调用

```java
// 1. 定义 Feign 接口
@FeignClient(name = "product-service",
             fallbackFactory = ProductFeignClientFallbackFactory.class)
public interface ProductFeignClient {

    @GetMapping("/api/products/{id}")
    ProductVO getProduct(@PathVariable Long id);

    @GetMapping("/api/products/batch")
    List<ProductVO> getProducts(@RequestParam List<Long> ids);

    @PostMapping("/api/products/{id}/stock/deduct")
    Boolean deductStock(@PathVariable Long id,
                        @RequestParam int quantity);
}

// 2. 降级处理
@Component
public class ProductFeignClientFallbackFactory
        implements FallbackFactory<ProductFeignClient> {

    @Override
    public ProductFeignClient create(Throwable cause) {
        return new ProductFeignClient() {
            @Override
            public ProductVO getProduct(Long id) {
                log.warn("Product service unavailable, id={}", id, cause);
                return ProductVO.empty(); // 返回空对象
            }

            @Override
            public List<ProductVO> getProducts(List<Long> ids) {
                return Collections.emptyList();
            }

            @Override
            public Boolean deductStock(Long id, int quantity) {
                return false; // 降级返回失败
            }
        };
    }
}
```

## 4.4 gRPC 服务通信

```protobuf
// product.proto
syntax = "proto3";

package com.example.product;

option java_package = "com.example.product.grpc";
option java_outer_classname = "ProductProto";

service ProductService {
    rpc GetProduct (GetProductRequest) returns (ProductResponse);
    rpc GetProductStream (GetProductsRequest)
        returns (stream ProductResponse);
    rpc DeductStock (DeductStockRequest) returns (DeductStockResponse);
}

message GetProductRequest {
    int64 id = 1;
}

message ProductResponse {
    int64 id = 1;
    string name = 2;
    double price = 3;
    int32 stock = 4;
    string status = 5;
```

## 5.3 JWT 认证过滤器

```java
@Component
@Slf4j
public class JwtAuthFilter implements GlobalFilter, Ordered {

    private static final String AUTH_HEADER = "Authorization";
    private static final String BEARER_PREFIX = "Bearer ";

    // 不需要认证的路径
    private static final Set<String> WHITE_LIST = Set.of(
        "/api/users/login",
        "/api/users/register",
        "/api/users/refresh-token",
        "/api/products",  // 公开商品列表
        "/actuator/health"
    );

    private final JwtTokenProvider tokenProvider;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                            GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 白名单放行
        if (WHITE_LIST.stream().anyMatch(path::startsWith)) {
            return chain.filter(exchange);
        }

        // 获取 Token
        String authHeader = exchange.getRequest()
            .getHeaders().getFirst(AUTH_HEADER);

        if (authHeader == null || !authHeader.startsWith(BEARER_PREFIX)) {
            return unauthorized(exchange, "Missing or invalid token");
        }

        String token = authHeader.substring(BEARER_PREFIX.length());

        try {
            // 验证 Token
            Claims claims = tokenProvider.parseToken(token);

            // 将用户信息注入请求头，传递给下游服务
            ServerWebExchange mutatedExchange = exchange.mutate()
                .request(exchange.getRequest().mutate()
                    .header("X-User-Id", claims.getSubject())
                    .header("X-User-Role", claims.get("role", String.class))
                    .header("X-User-Name", claims.get("name", String.class))
                    .build())
                .build();

            return chain.filter(mutatedExchange);

        } catch (ExpiredJwtException e) {
            return unauthorized(exchange, "Token expired");
        } catch (JwtException e) {
            return unauthorized(exchange, "Invalid token");
        }
    }

    private Mono<Void> unauthorized(ServerWebExchange exchange, String message) {
        exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
        exchange.getResponse().getHeaders()
            .setContentType(MediaType.APPLICATION_JSON);
        String body = String.format("{\"code\":401,\"message\":\"%s\"}", message);
        DataBuffer buffer = exchange.getResponse().bufferFactory()
            .wrap(body.getBytes(StandardCharsets.UTF_8));
        return exchange.getResponse().writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() {
        return -100; // 最高优先级执行
    }
}
```

# Part 7: 分布式事务

## 7.1 分布式事务的挑战

```
在微服务架构中，一个业务操作可能涉及多个服务的数据库:

  下单流程:
    1. 订单服务: 创建订单 (order_db)
    2. 库存服务: 扣减库存 (inventory_db)
    3. 支付服务: 创建支付单 (payment_db)
    4. 账户服务: 冻结余额 (account_db)

  问题:
    步骤1成功，步骤2失败 --> 订单已创建但库存未扣减（数据不一致）
    步骤3超时 --> 不知道是否成功（需要对账）
    如何回滚已成功的步骤？
```

## 7.2 Saga 模式

```
Saga 是一系列本地事务的序列，每个本地事务由一个服务执行。
如果某一步失败，通过补偿事务（Compensating Transaction）回滚之前的操作。

两种实现方式:

  1. 编排式 Saga (Choreography)
     每个服务完成后发布事件，下一个服务监听事件执行

  +--------+    event    +----------+    event    +--------+
  | 订单   |----------->| 库存服务  |----------->| 支付   |
  | 服务   |            |           |            | 服务   |
  |        |<-----------| (失败时)  |            |        |
  |        |  补偿事件  |           |            |        |
  +--------+            +----------+            +--------+

  2. 协调式 Saga (Orchestration)
     Saga Orchestrator 负责协调所有步骤

  +----------------+
  |  Saga 协调者   |
  +-------+--------+
          |
   +------+------+
   |      |      |
  订单  库存   支付
  服务  服务   服务
```

```java
// 编排式 Saga 实现示例
@Component
@Slf4j
public class CreateOrderSaga {

    private final KafkaTemplate<String, String> kafka;
    private final ObjectMapper mapper;

    // Step 1: 创建订单后发布事件
    @Transactional
    public Order createOrder(CreateOrderCommand command) {
        Order order = new Order(command);
        order.setPending();
        orderRepository.save(order);

        // 发布事件触发库存扣减
        publishEvent("inventory.reserve", new ReserveStockEvent(
            order.getId(), order.getItems()));

        return order;
    }

    // 监听库存扣减结果
    @KafkaListener(topics = "inventory.reserved")
    public void onStockReserved(ConsumerRecord<String, String> record) {
        StockReservedEvent event = parseEvent(record.value());
        Order order = orderRepository.findById(event.getOrderId()).orElseThrow();

        if (event.isSuccess()) {
            // 发布事件触发支付
            publishEvent("payment.create", new CreatePaymentEvent(
                order.getId(), order.getTotalAmount()));
        } else {
            // 库存不足，取消订单
            order.cancel("库存不足");
            orderRepository.save(order);
        }
    }

    // 监听支付结果
    @KafkaListener(topics = "payment.completed")
    public void onPaymentCompleted(ConsumerRecord<String, String> record) {
        PaymentCompletedEvent event = parseEvent(record.value());
        Order order = orderRepository.findById(event.getOrderId()).orElseThrow();

        if (event.isSuccess()) {
            order.confirm();
            // 发布订单确认事件（通知物流等服务）
            publishEvent("order.confirmed",
                new OrderConfirmedEvent(order.getId()));
        } else {
            order.cancel("支付失败");
            // 补偿：归还库存
            publishEvent("inventory.release",
                new ReleaseStockEvent(order.getId(), order.getItems()));
            orderRepository.save(order);
        }
    }
}
```

## 7.4 本地消息表（Outbox Pattern）

```java
// 发件箱模式：保证消息可靠投递
// 原理：将消息写入本地数据库，与业务操作在同一事务中
//       异步轮询发件箱表，将消息投递到 Kafka

@Entity
@Table(name = "outbox_events")
@Data
public class OutboxEvent {
    @Id @GeneratedValue
    private Long id;
    private String aggregateType; // 如 "Order"
    private String aggregateId;   // 如 orderId
    private String eventType;     // 如 "order.created"
    private String payload;       // JSON 事件内容
    private String topic;         // Kafka topic
    private LocalDateTime createdAt;
    private boolean published = false;
}

@Service
@Transactional
public class OrderService {

    private final OrderRepository orderRepo;
    private final OutboxEventRepository outboxRepo;

    public Order createOrder(CreateOrderRequest req) {
        // 1. 创建订单
        Order order = new Order(req);
        orderRepo.save(order);

        // 2. 同一事务内写入发件箱
        OutboxEvent outboxEvent = OutboxEvent.builder()
            .aggregateType("Order")
            .aggregateId(order.getId().toString())
            .eventType("order.created")
            .topic("order.created")
            .payload(toJson(order))
            .createdAt(LocalDateTime.now())
            .build();
        outboxRepo.save(outboxEvent);

        // 事务提交后，轮询任务会自动将事件投递到 Kafka
        return order;
    }
}

@Component
@Slf4j
public class OutboxEventPublisher {

    private final OutboxEventRepository outboxRepo;
    private final KafkaTemplate<String, String> kafka;

    @Scheduled(fixedDelay = 1000) // 每秒轮询
    @Transactional
    public void publishPendingEvents() {
        List<OutboxEvent> events = outboxRepo
            .findByPublishedFalseOrderByCreatedAtAsc();

        for (OutboxEvent event : events) {
            try {
                kafka.send(event.getTopic(),
                    event.getAggregateId(), event.getPayload()).get();
                event.setPublished(true);
                outboxRepo.save(event);
            } catch (Exception e) {
                log.error("Failed to publish outbox event: {}", event.getId(), e);
            }
        }
    }
}
```

```java
@Component
public class JwtTokenProvider {

    @Value("${jwt.secret}")
    private String secretKey;

    @Value("${jwt.expiration:3600000}")
    private long expirationMs;  // 1小时

    @Value("${jwt.refresh-expiration:604800000}")
    private long refreshExpirationMs;  // 7天

    private SecretKey getSigningKey() {
        byte[] keyBytes = Decoders.BASE64.decode(secretKey);
        return Keys.hmacShaKeyFor(keyBytes);
    }

    // 生成 Access Token
    public String generateToken(UserDetails user) {
        return Jwts.builder()
            .subject(user.getUsername())
            .claim("roles", user.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority)
                .collect(Collectors.toList()))
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expirationMs))
            .signWith(getSigningKey())
            .compact();
    }

    // 解析 Token
    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    public boolean isTokenValid(String token, String username) {
        final Claims claims = parseToken(token);
        return claims.getSubject().equals(username) &&
               !claims.getExpiration().before(new Date());
    }
}
```

# Part 12: 微服务反模式

## 12.1 分布式单体（Distributed Monolith）

```
症状:
  - 服务间通过共享数据库直接关联
  - 任何服务的改动都需要同时部署其他服务
  - 服务间存在循环依赖
  - 运行时服务间高度耦合

危害:
  享有了微服务的所有复杂性，却没有得到任何好处

解决方案:
  - 严格遵守 Database Per Service
  - 用事件驱动替代直接数据库关联
  - 通过 API 而非数据库共享数据
```

## 12.2 过度拆分（Nano Services）

```
症状:
  - 每个服务只有几百行代码
  - 一个业务操作需要调用 10+ 个服务
  - 大量分布式事务
  - 大量网络调用导致延迟高

解决方案:
  - 按 DDD 限界上下文重新划分服务边界
  - 合并高度耦合的服务
  - 遵循"2 Pizza Team"原则
```

## 12.3 同步调用链（Synchronous Call Chain）

```
症状:
  A --> B --> C --> D --> E
  总延迟 = La + Lb + Lc + Ld + Le
  任何一个服务故障，整个链路失败

解决方案:
  1. 异步化：用消息队列替代同步调用
  2. 并行化：不相互依赖的调用并发执行
  3. 缓存：减少对后端服务的直接调用
  4. 熔断器：防止雪崩效应
```

## 12.4 无版本 API（Versioning Neglect）

```
症状:
  - API 修改导致其他服务无法工作
  - 所有服务必须同时升级（协调部署）
  - API 向后不兼容

最佳实践:
  1. URL 版本化: /api/v1/orders, /api/v2/orders
  2. 消费者驱动的契约测试（Pact）
  3. Postel 法则: 对接收的内容宽容，对发送的内容严格
  4. 先添加新字段（向后兼容），再废弃旧字段
  5. 保持至少两个版本并行运行
```

## 12.5 日志孤岛（Log Islands）

```
症状:
  - 每个服务的日志只在本地，没有集中收集
  - 出问题时需要登录每台服务器查日志
  - 跨服务排查问题极其困难

解决方案:
  1. ELK Stack: Filebeat --> Logstash --> Elasticsearch --> Kibana
  2. Loki + Grafana（轻量级方案）
  3. 统一日志格式（JSON）
  4. 请求链路 ID 贯穿所有日志
  5. 分布式链路追踪（SkyWalking/Jaeger）
```

## 13.6 Redis 缓存设计

```java
// 商品缓存策略
@Service
@Slf4j
public class ProductCacheService {

    private final RedisTemplate<String, Object> redisTemplate;
    private final ProductRepository productRepository;

    private static final String PRODUCT_KEY = "product:detail:";
    private static final String PRODUCT_LIST_KEY = "product:category:";
    private static final long PRODUCT_TTL = 3600;  // 1小时
    private static final long NULL_VALUE_TTL = 60; // 防缓存穿透

    public ProductVO getProduct(Long id) {
        String key = PRODUCT_KEY + id;

        // 1. 先查缓存
        Object cached = redisTemplate.opsForValue().get(key);
        if (cached != null) {
            if (cached instanceof String && cached.equals("NULL")) {
                return null; // 缓存空值，防穿透
            }
            return (ProductVO) cached;
        }

        // 2. 缓存未命中，查数据库（加分布式锁防击穿）
        String lockKey = "product:lock:" + id;
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

        try {
            if (Boolean.TRUE.equals(locked)) {
                ProductVO product = productRepository.findById(id)
                    .map(this::toVO)
                    .orElse(null);

                if (product == null) {
                    // 缓存空值，防止缓存穿透
                    redisTemplate.opsForValue().set(
                        key, "NULL", NULL_VALUE_TTL, TimeUnit.SECONDS);
                } else {
                    // 设置随机过期时间，防止缓存雪崩
                    long ttl = PRODUCT_TTL + (long)(Math.random() * 600);
                    redisTemplate.opsForValue().set(
                        key, product, ttl, TimeUnit.SECONDS);
                }
                return product;
            } else {
                // 等待其他线程设置缓存
                Thread.sleep(100);
                return getProduct(id);  // 重试
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return productRepository.findById(id).map(this::toVO).orElse(null);
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
}
```

## Q1：微服务拆分的粒度如何把握？

```
没有统一标准，但有以下参考依据:

  1. 团队规模法则（2 Pizza Team）
     一个服务的团队规模应该在 6~10 人，能被 2 个披萨喂饱

     一个限界上下文对应一个微服务

  3. 变更频率法则
     变更频率差异大的模块应该独立成服务

  4. 扩展需求法则
     有差异化扩展需求的模块独立成服务

  5. 代码量参考
     一个微服务通常 5000~50000 行代码
     少于 1000 行可能是 Nano Service
     超过 100000 行可能是 Mini Monolith
```

## Q2：如何保证微服务间的数据一致性？

```
策略1: 拥抱最终一致性
  - 大多数业务场景可以接受短暂不一致（几秒到几分钟）
  - 通过消息队列 + 消费幂等 + 重试机制保证最终一致

策略2: Saga 模式
  - 将跨服务事务拆分为本地事务 + 补偿事务序列
  - 编排式 Saga: 中心化协调
  - 编舞式 Saga: 去中心化事件驱动

策略3: 2PC/Seata AT 模式
  - 适用于对一致性要求较高的场景
  - 性能比 Saga 差，但一致性更强

策略4: 只读跨服务查询
  - 通过 API 组合（BFF）从多个服务获取数据
  - CQRS + 物化视图：将查询数据冗余到专门的读模型
```

## Q3：服务注册与发现的原理是什么？

```
客户端发现模式（Client-Side Discovery）:

  1. 服务启动时向注册中心注册（服务名、IP、端口、健康检查URL）
  2. 注册中心通过心跳检测服务健康状态
  3. 客户端调用前先向注册中心查询可用实例列表
  4. 客户端通过负载均衡算法选择一个实例
  5. 直接调用选中的实例

  优点: 客户端可以实现复杂的负载均衡策略（区域感知、权重等）
  缺点: 客户端需要内置服务发现逻辑

服务端发现模式（Server-Side Discovery）:

  1. 请求先到达负载均衡器
  2. 负载均衡器查询注册中心获取实例列表
  3. 负载均衡器选择实例并转发请求

  优点: 客户端无需关心服务发现
  代表: Kubernetes Service, Nginx + Consul
```

## Q4：什么是 CAP 理论？微服务如何权衡？

```
CAP 理论:
  C (Consistency):    一致性 - 所有节点同时看到相同数据
  A (Availability):   可用性 - 每个请求都能收到响应
  P (Partition Tolerance): 分区容错性 - 网络分区时仍可运行

  分布式系统中，P 是必须的（网络故障不可避免），
  因此只能在 C 和 A 之间做取舍:

  CP 系统: 优先一致性（ZooKeeper, HBase, etcd）
  AP 系统: 优先可用性（Cassandra, Nacos(默认), CouchDB）

微服务场景中的权衡:
  - 注册中心: Nacos 默认 AP（可以切换为 CP 模式）
  - 数据库: MySQL 是 CP（强一致性），NoSQL 多为 AP
  - 业务层面: 大多数业务接受最终一致性（AP + 补偿）
  - 金融场景: 需要 CP，但会牺牲部分可用性
```

## Q5：如何设计幂等的接口？

```java
// 幂等设计方案：基于唯一幂等键 + Redis 防重
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final RedisTemplate<String, String> redisTemplate;
    private final OrderService orderService;

    @PostMapping
    public Result<OrderVO> createOrder(
            @RequestHeader("X-Idempotency-Key") String idempotencyKey,
            @RequestBody CreateOrderRequest req) {

        String lockKey = "idempotency:" + idempotencyKey;

        // 检查是否已处理过
        String existingResult = redisTemplate.opsForValue().get(lockKey);
        if (existingResult != null) {
            // 返回之前的处理结果（幂等性）
            return Result.ok(parseOrderVO(existingResult));
        }

        // 占位，防止并发重复处理
        Boolean set = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "PROCESSING", 10, TimeUnit.MINUTES);

        if (!Boolean.TRUE.equals(set)) {
            // 并发请求，等待
            return Result.fail(409, "请求处理中，请勿重复提交");
        }

        try {
            OrderVO order = orderService.createOrder(req);
            // 保存结果
            redisTemplate.opsForValue().set(
                lockKey, toJson(order), 24, TimeUnit.HOURS);
            return Result.ok(order);
        } catch (Exception e) {
            redisTemplate.delete(lockKey);
            throw e;
        }
    }
}
```

## Q6：服务雪崩是什么？如何防止？

```
雪崩效应 (Cascading Failure):

  A --> B --> C --> D

  D 服务变慢 --> C 的线程池耗尽 --> B 的线程池耗尽 --> A 不可用

防止雪崩的手段:

  1. 超时控制
     - Feign/RestTemplate 设置合理的超时时间（建议 2~5 秒）
     - 防止线程长时间阻塞

  2. 舱壁隔离（Bulkhead）
     - 每个下游服务使用独立线程池
     - D 挂了不会影响其他调用路径

  3. 熔断器（Circuit Breaker）
     - 错误率超阈值后，快速失败（不等待）
     - Sentinel / Resilience4j 实现

  4. 限流（Rate Limiting）
     - 在网关层或服务层限制请求速率
     - 保护后端不被流量冲垮

  5. 降级（Fallback）
     - 主路径失败时返回缓存数据/默认值
     - 保证核心功能可用
```

## Q7：如何实现微服务的灰度发布？

```
灰度发布 (Canary Release) 实现方案:

  方案1: Kubernetes + Ingress 权重
    部署两个版本的 Deployment（v1, v2）
    通过 Ingress canary 注解控制流量比例

  方案2: Istio 服务网格
    通过 VirtualService 定义流量规则
    支持按请求头、用户属性、百分比路由

  方案3: Spring Cloud Gateway + Nacos 元数据
    给新版本实例打标签 version=canary
    在 Gateway 中读取请求头，路由到对应版本

  方案4: 特性开关（Feature Toggle）
    在代码中通过配置开关控制新功能的开启
    可以按用户ID、A/B 测试分组等维度控制
```

## Q8：什么是服务网格（Service Mesh）？和微服务框架有什么区别？

```
服务网格（Service Mesh）:
  在基础设施层面解决服务间通信问题，与业务代码完全无关

  实现方式: Sidecar 模式
  每个服务 Pod 旁边部署一个 Envoy Proxy（Sidecar）
  所有入/出流量都经过 Sidecar

  +----------+     +----------+
  | 服务 A   |     | 服务 B   |
  | +------+ |     | +------+ |
  | | Envoy| |---->| | Envoy| |
  | +------+ |     | +------+ |
  +----------+     +----------+
        \              /
         Istio Control Plane

功能对比:
+------------------+------------------+------------------+
| 功能             | Spring Cloud     | Istio            |
+------------------+------------------+------------------+
| 负载均衡         | 代码中实现        | Sidecar 实现     |
| 熔断             | Sentinel/R4j      | Envoy 实现       |
| 链路追踪         | SkyWalking Agent  | 自动注入         |
| mTLS             | 手动配置           | 自动             |
| 语言无关         | Java 专属         | 语言无关         |
+------------------+------------------+------------------+
```

## Q9：微服务的测试策略是什么？

```
测试金字塔（从下到上）:

                    /\
                   /  \
                  / E2E\
                 /  测试 \
                /--------\
               / 集成测试  \
              /------------\
             /  单元测试    \
            /________________\

  1. 单元测试 (Unit Test) - 占70%
     测试单个类/方法，Mock 外部依赖

  2. 集成测试 (Integration Test) - 占20%
     测试服务与数据库/Redis/消息队列的集成
     使用 Testcontainers 启动真实的依赖组件

  3. 契约测试 (Contract Test)
     Consumer-Driven Contract Testing（Pact）
     验证服务间 API 的契约兼容性

  4. E2E 测试 (End-to-End Test) - 占10%
     模拟真实用户操作，验证完整业务流程

  5. 混沌工程 (Chaos Engineering)
     主动注入故障，验证系统弹性
     工具：Chaos Monkey, ChaosBlade
```

## Q10：微服务的配置管理最佳实践有哪些？

```
12-Factor App 配置原则:

  1. 环境无关的配置: 代码中不能有环境相关的硬编码
  2. 配置从环境变量读取或配置中心读取
  3. 敏感配置加密（不能出现在代码仓库）

配置分类管理:

  | 配置类型         | 存储位置               | 更新方式       |
  |-----------------|------------------------|---------------|
  | 服务基础配置    | Nacos / Config Server   | 动态推送       |
  | 数据库连接      | K8s Secret / Vault      | 重启生效       |
  | 功能开关        | 配置中心（动态）         | 实时生效       |
  | 业务规则        | 数据库 + 缓存           | 实时生效       |
  | 应用启动参数    | 环境变量                 | 重启生效       |

最佳实践:
  1. 使用 @ConfigurationProperties + @RefreshScope 实现动态刷新
  2. 多环境配置文件隔离（dev/test/staging/prod）
  3. 配置版本化，支持回滚
  4. 监控配置变更，异常时告警
  5. 禁止在代码中 hardcode 任何环境相关值
```
