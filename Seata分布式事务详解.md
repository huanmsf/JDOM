# Part 1: 分布式事务背景

## 1.1 本地事务 vs 分布式事务

### 1.1.1 本地事务

本地事务（Local Transaction）是指在单个数据库连接范围内执行的事务。所有的数据库操作都在同一个数据库实例上完成，由数据库的事务管理器全权负责，保证 ACID 特性。

```
+---------------------------------------------+
|              单体应用架构                     |
|                                             |
|   +----------+       +-----------------+   |
|   |  业务层   |------>|   数据库         |   |
|   | Service  |       |  MySQL/Oracle   |   |
|   +----------+       |                 |   |
|                      |  表A: 账户       |   |
|                      |  表B: 订单       |   |
|                      |  表C: 库存       |   |
|                      +-----------------+   |
|                                             |
|  事务范围：同一个数据库连接，由数据库保证ACID   |
+---------------------------------------------+
```

**本地事务的 ACID 特性：**

| 特性 | 英文 | 说明 |
|------|------|------|
| 原子性 | Atomicity | 事务内所有操作要么全部成功，要么全部失败回滚 |
| 一致性 | Consistency | 事务前后数据满足完整性约束 |
| 隔离性 | Isolation | 并发事务互相隔离，不相互干扰 |
| 持久性 | Durability | 事务提交后，数据永久保存 |

本地事务示例代码：

```java
// 本地事务示例
@Transactional
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    // 1. 扣减转出方余额
    accountMapper.deductBalance(fromId, amount);
    // 2. 增加转入方余额
    accountMapper.addBalance(toId, amount);
    // 两个操作在同一个数据库连接中，由 Spring + 数据库保证原子性
}
```

### 1.1.2 分布式事务

分布式事务（Distributed Transaction）是指涉及多个独立数据库、多个服务节点的事务。每个参与者有自己的本地事务，但需要在逻辑上保证所有参与者的操作要么全部提交，要么全部回滚。

```
+-----------------------------------------------------------------+
|                        微服务架构                                 |
|                                                                 |
|  +-------------+    +-------------+    +-------------+         |
|  |  订单服务    |    |  库存服务    |    |  账户服务    |         |
|  |             |    |             |    |             |         |
|  |  +-------+  |    |  +-------+  |    |  +-------+  |         |
|  |  | 订单DB |  |    |  | 库存DB |  |    |  | 账户DB |  |         |
|  |  +-------+  |    |  +-------+  |    |  +-------+  |         |
|  +-------------+    +-------------+    +-------------+         |
|                                                                 |
|  问题：三个数据库分散在不同节点，如何保证事务的原子性？              |
+-----------------------------------------------------------------+
```

**分布式事务面临的核心挑战：**

1. **网络不可靠**：服务间通信通过网络，网络可能延迟、丢包、分区
2. **节点可能宕机**：任何一个服务节点都可能在任何时刻崩溃
3. **无全局时钟**：分布式系统中没有全局统一的时钟，无法精确排序事件
4. **部分失败**：系统可能处于部分成功、部分失败的中间状态

## 1.2 分布式事务产生的原因（微服务拆库）

### 1.2.1 微服务化拆分的背景

随着业务规模的增长，单体应用逐渐暴露出以下问题：

- **代码耦合严重**：所有业务逻辑堆积在一个工程中，模块间耦合度极高
- **部署风险高**：修改一个小功能，需要重新部署整个应用，风险极大
- **技术栈固化**：无法针对不同模块选择最合适的技术栈
- **水平扩展困难**：无法针对单个高负载模块进行独立扩缩容

### 1.2.2 微服务拆库带来分布式事务问题

```
+-----------------------------------------------------------------------+
|                    电商系统 - 微服务架构演进                               |
|                                                                       |
|  +------------------------------------------+                        |
|  |              单体架构（之前）               |                        |
|  |   +--------------------------------+    |                        |
|  |   |          MySQL 数据库           |    |                        |
|  |   |  order表 | stock表 | account表  |    |                        |
|  |   +--------------------------------+    |                        |
|  |   @Transactional 一个注解解决所有问题    |                        |
|  +------------------------------------------+                        |
|                          | 拆分                                        |
|                          v                                            |
|  +----------------------------------------------------------------+   |
|  |              微服务架构（之后）                                  |   |
|  |                                                                |   |
|  |  +--------------+  +--------------+  +--------------+         |   |
|  |  |   订单服务     |  |   库存服务    |  |   账户服务    |         |   |
|  |  |  order_db    |  |  stock_db    |  |  account_db  |         |   |
|  |  |  192.168.1.1 |  |  192.168.1.2 |  |  192.168.1.3 |         |   |
|  |  +--------------+  +--------------+  +--------------+         |   |
|  |                                                                |   |
|  |  问题：用户下单时，需要同时操作3个数据库，@Transactional 失效！      |   |
|  +----------------------------------------------------------------+   |
+-----------------------------------------------------------------------+
```

## 1.3 CAP 理论详解

CAP 理论由 Eric Brewer 在 2000 年提出，是分布式系统设计的基础理论之一。它指出在一个分布式系统中，**一致性（Consistency）**、**可用性（Availability）** 和 **分区容错性（Partition Tolerance）** 三者无法同时满足，最多只能同时满足其中两个。

```
                    CAP 三角形

                    一致性 (C)
                       /\
                      /  \
                     /    \
                    /  CAP \
                   /        \
                  /----------\
         可用性 (A)            分区容错性 (P)

+----------------------------------------------------------+
|                                                          |
|  CA：满足一致性和可用性，放弃分区容错性                       |
|      -> 传统关系型数据库（单机MySQL），不适合分布式场景          |
|                                                          |
|  CP：满足一致性和分区容错性，放弃可用性                       |
|      -> ZooKeeper、HBase、Consul                          |
|      -> 发生网络分区时，系统选择停服来保证一致性               |
|                                                          |
|  AP：满足可用性和分区容错性，放弃强一致性                      |
|      -> Cassandra、Eureka、CouchDB                        |
|      -> 发生网络分区时，仍然可用，但可能返回过期数据            |
|                                                          |
+----------------------------------------------------------+
```

### 1.3.1 一致性（Consistency）

所有节点在同一时间访问到的数据是相同的。即对系统的任何读操作，都能读取到最新写入的数据，或者返回错误。

### 1.3.2 可用性（Availability）

系统中非故障节点收到的每个请求都必须得到响应（不能返回错误，但可以不是最新数据）。

### 1.3.3 分区容错性（Partition Tolerance）

即使网络发生分区（部分节点间无法通信），系统仍然继续运行。

**重要结论：在真实的分布式系统中，网络分区是不可避免的，因此 P 必须被满足，实际上只有 CP 和 AP 两种选择。**

## 1.4 BASE 理论

BASE 理论是 CAP 理论的延伸，是对 AP 系统的实践总结，由 eBay 架构师 Dan Pritchett 在 2008 年提出。BASE 代表：

- **BA**sically Available（基本可用）
- **S**oft state（软状态）
- **E**ventually consistent（最终一致性）

### 1.4.1 基本可用（Basically Available）

系统在出现故障时，允许损失部分可用性，保证核心功能可用。

```
示例：
+---------------------------------------------------------+
|  正常情况：搜索响应时间 < 0.5 秒                            |
|  故障时：搜索响应时间 < 3 秒（降级处理，保证基本可用）          |
|                                                         |
|  正常情况：所有商品信息完整展示                              |
|  故障时：核心商品信息展示，评论等非核心功能不可用（熔断降级）      |
+---------------------------------------------------------+
```

### 1.4.2 软状态（Soft State）

允许系统在不同节点之间存在数据不一致的中间状态，即允许数据同步存在延时。

```
软状态示意图：

  时间轴:  t0          t1          t2          t3
           |           |           |           |
  节点A:   X=1 --------------------------------------------> X=1
  节点B:   X=0 ------ 同步中 --------------------------------> X=1
  节点C:   X=0 --------------- 同步中 ---------------------> X=1

  t1~t2 之间：系统处于软状态（数据不一致的中间状态）
  t3 之后：达到最终一致性
```

### 1.4.3 最终一致性（Eventually Consistent）

系统中所有数据副本，经过一段时间的同步后，最终能够达到一致的状态。

最终一致性的几种变体：

| 变体 | 说明 |
|------|------|
| 因果一致性 | 有因果关系的操作按因果顺序执行 |
| 读己之写一致性 | 自己写入的数据，自己总能读到 |
| 会话一致性 | 在同一会话内保证读己之写 |
| 单调读一致性 | 一旦读到某个值，后续读操作不会读到更旧的值 |
| 单调写一致性 | 同一客户端的写操作按顺序执行 |

## 1.5 ACID vs BASE 对比

```
+--------------+------------------------------+------------------------------+
|    维度       |         ACID                 |         BASE                 |
+--------------+------------------------------+------------------------------+
|  一致性要求   | 强一致性                       | 弱一致性（最终一致性）           |
+--------------+------------------------------+------------------------------+
|  可用性       | 为保证强一致性，可能牺牲可用性   | 优先保证可用性                  |
+--------------+------------------------------+------------------------------+
|  隔离性       | 强隔离（读锁/写锁/MVCC）        | 弱隔离，允许脏读等              |
+--------------+------------------------------+------------------------------+
|  应用场景     | 银行转账、财务系统               | 电商、社交媒体、推荐系统         |
+--------------+------------------------------+------------------------------+
|  代表技术     | MySQL/Oracle（传统RDBMS）      | Cassandra/MongoDB/Redis       |
+--------------+------------------------------+------------------------------+
|  扩展性       | 垂直扩展为主，水平扩展困难        | 水平扩展，天然分布式             |
+--------------+------------------------------+------------------------------+
```

# Part 2: 传统分布式事务方案详解

## 2.1 两阶段提交（2PC）

### 2.1.1 概述

两阶段提交（Two-Phase Commit，2PC）是最经典的分布式事务协议，由协调者（Coordinator）和参与者（Participant）组成。整个事务分为两个阶段：**准备阶段（Prepare Phase）** 和 **提交/回滚阶段（Commit/Rollback Phase）**。

### 2.1.2 两阶段提交流程详解

**第一阶段：准备阶段（Prepare Phase）**

```
+-------------------------------------------------------------------+
|                     2PC - 第一阶段（Prepare）                       |
|                                                                   |
|   协调者（Coordinator）              参与者（Participants）          |
|         |                        A        B        C             |
|         |                        |        |        |             |
|         |---- PREPARE ---------->|        |        |             |
|         |---- PREPARE ---------->|        |        |             |
|         |---- PREPARE ----------------------->|    |             |
|         |---- PREPARE --------------------------------->|         |
|         |                        |        |        |             |
|         |    各参与者执行本地事务，写redo/undo log，但不提交           |
|         |                        |        |        |             |
|         |<--- YES --------------- A        |        |             |
|         |<--- YES ------------------------------ B  |             |
|         |<--- YES ----------------------------------- C            |
|         |                                                        |
|         |   协调者收到所有参与者的 YES 响应，进入第二阶段               |
+-------------------------------------------------------------------+
```

**第二阶段-提交：所有参与者都返回 YES**

```
+-------------------------------------------------------------------+
|                 2PC - 第二阶段（Commit）- 全部成功                   |
|                                                                   |
|   协调者（Coordinator）              参与者（Participants）          |
|         |                        A        B        C             |
|         |---- COMMIT ----------->|        |        |             |
|         |---- COMMIT ----------------------->|      |             |
|         |---- COMMIT --------------------------------->|           |
|         |                        |        |        |             |
|         |          各参与者提交本地事务，释放资源                      |
|         |                                                        |
|         |<--- ACK --------------- A                               |
|         |<--- ACK ------------------------------ B                |
|         |<--- ACK ----------------------------------- C            |
|         |                                                        |
|         |           全局事务提交成功！                                |
+-------------------------------------------------------------------+
```

**第二阶段-回滚：任一参与者返回 NO 或超时**

```
+-------------------------------------------------------------------+
|                 2PC - 第二阶段（Rollback）- 某方失败                 |
|                                                                   |
|   协调者（Coordinator）              参与者（Participants）          |
|         |                        A        B        C             |
|         |<--- YES --------------- A                               |
|         |<--- YES ------------------------------ B                |
|         |<--- NO  ----------------------------------- C (C失败)    |
|         |                                                        |
|         |   协调者收到 NO，决定回滚                                   |
|         |                                                        |
|         |---- ROLLBACK ---------->|        |        |            |
|         |---- ROLLBACK ------------------>|         |            |
|         |---- ROLLBACK ---------------------------->|            |
|         |                                                        |
|         |          所有参与者执行回滚，释放锁资源                       |
+-------------------------------------------------------------------+
```

### 2.1.3 2PC 存在的问题

**问题1：同步阻塞（Synchronous Blocking）**

```
+-----------------------------------------------------------------+
|                    同步阻塞问题                                    |
|                                                                 |
|  从 Prepare 开始，到 Commit/Rollback 结束，所有参与者都持有锁         |
|                                                                 |
|  时间轴:                                                         |
|  参与者A: [锁定资源]=========================[释放资源]            |
|  参与者B: [锁定资源]=========================[释放资源]            |
|  参与者C: [锁定资源]=========================[释放资源]            |
|                                                                 |
|  在整个 Prepare -> Commit 过程中，锁一直被持有，                    |
|  其他事务无法访问这些资源，性能极差！                                |
+-----------------------------------------------------------------+
```

**问题2：单点故障（Single Point of Failure）**

```
+-----------------------------------------------------------------+
|                    单点故障问题                                    |
|                                                                 |
|  场景：协调者在发送 COMMIT 命令给 A 后崩溃                          |
|                                                                 |
|   协调者（已宕机！）         参与者 A         B         C         |
|         |---- COMMIT ------->A  (A 已提交)                      |
|         | [崩溃！]                                               |
|         X                         B: 等待响应...陷入阻塞！        |
|    无法继续发送命令                  C: 等待响应...陷入阻塞！        |
|                                                                 |
|  B 和 C 不知道该提交还是回滚，只能等待协调者恢复，期间持续持有锁！      |
+-----------------------------------------------------------------+
```

**问题3：数据不一致**

```
+-----------------------------------------------------------------+
|                    数据不一致问题                                  |
|                                                                 |
|  场景：协调者发送 Commit 后，部分参与者收到，部分因网络问题未收到        |
|                                                                 |
|   协调者                    A         B         C              |
|         |---- COMMIT ------>A (提交)                           |
|         |---- COMMIT ---X  (网络故障，B未收到)                   |
|         |---- COMMIT --------------------------------->C (提交)  |
|         |                                                       |
|  结果：A 和 C 提交了，B 没有提交，数据永久不一致！                     |
+-----------------------------------------------------------------+
```

**2PC 问题汇总：**

| 问题 | 描述 | 影响 |
|------|------|------|
| 同步阻塞 | 参与者在等待协调者响应期间，一直持有锁 | 性能极差 |
| 单点故障 | 协调者是单点，崩溃导致整个事务悬挂 | 可用性差 |
| 数据不一致 | 协调者崩溃时，部分参与者已提交，部分未提交 | 数据错误 |
| 脑裂问题 | 网络分区导致参与者和协调者决策不一致 | 数据混乱 |

## 2.2 三阶段提交（3PC）

### 2.2.1 3PC 概述

三阶段提交（Three-Phase Commit，3PC）是对 2PC 的改进，将 2PC 的第一阶段拆分为两个阶段，共三个阶段：

1. **CanCommit**：询问阶段，不锁定资源，只询问参与者是否可以提交
2. **PreCommit**：预提交阶段，锁定资源，执行事务但不提交
3. **DoCommit**：提交阶段，正式提交或回滚

### 2.2.2 3PC 完整流程

```
+-------------------------------------------------------------------+
|                   3PC - 三阶段提交完整流程                           |
|                                                                   |
|  ========= 第一阶段：CanCommit (不锁资源，仅询问) =========          |
|                                                                   |
|   协调者                          参与者 A, B, C                  |
|     |---- CanCommit? -----------------> A                         |
|     |---- CanCommit? -----------------------> B                   |
|     |---- CanCommit? ---------------------------------> C          |
|     |<--- YES ----------------------- A                           |
|     |<--- YES ------------------------------- B                   |
|     |<--- YES --------------------------------------- C           |
|                                                                   |
|  ========= 第二阶段：PreCommit (锁资源，执行事务) =========           |
|                                                                   |
|     |---- PreCommit -----------------> A  (执行事务，锁定资源)      |
|     |---- PreCommit -----------------------> B  (执行事务)         |
|     |---- PreCommit ---------------------------------> C           |
|     |<--- ACK ----------------------- A                           |
|     |<--- ACK ------------------------------- B                   |
|     |<--- ACK --------------------------------------- C           |
|                                                                   |
|  ========= 第三阶段：DoCommit (正式提交或回滚) =========              |
|                                                                   |
|     |---- DoCommit ----------------> A  (正式提交)                 |
|     |---- DoCommit --------------------> B  (正式提交)              |
|     |---- DoCommit ---------------------------------> C  (提交)    |
|     |<--- ACK ----------------------- A                           |
|     |<--- ACK ------------------------------- B                   |
|     |<--- ACK --------------------------------------- C           |
|                                                                   |
|                      事务成功提交！                                  |
+-------------------------------------------------------------------+
```

### 2.2.3 3PC 对 2PC 的改进

| 改进点 | 2PC | 3PC |
|--------|-----|-----|
| 阻塞问题 | 第一阶段直接锁资源 | 增加 CanCommit 阶段，不锁资源 |
| 单点故障 | 协调者宕机后参与者无限等待 | 参与者超时后可以自行决定提交 |
| 超时机制 | 无超时处理 | 各阶段引入超时机制，超时自行决策 |

### 2.2.4 3PC 仍然存在的问题

```
+-----------------------------------------------------------------+
|              3PC 仍然存在的问题：网络分区导致数据不一致              |
|                                                                 |
|  场景：进入 DoCommit 阶段后发生网络分区                            |
|                                                                 |
|   协调者                    A           B           C          |
|         |--DoCommit------->| (收到，提交)              |          |
|         |--DoCommit------------------>| (收到，提交)   |          |
|         |--DoCommit---- 网络分区 ---> C 未收到         |          |
|         |                             | C 等待超时...  |          |
|         |                             | 超时后自行提交！|          |
|                                                                 |
|  如果协调者在超时前决定回滚（因其他原因），然后通知A、B回滚：            |
|    A、B 回滚，C 超时后自行提交，数据不一致！                         |
+-----------------------------------------------------------------+
```

**结论：3PC 无法完全解决分布式事务的一致性问题，只是在工程上减少了阻塞时间。**

## 2.3 本地消息表

### 2.3.1 原理概述

本地消息表（Local Message Table）是一种基于消息中间件的最终一致性方案。核心思想是：**将跨服务的分布式事务转化为本地事务 + 消息重试**。

### 2.3.2 本地消息表完整流程

```
+-----------------------------------------------------------------------+
|                      本地消息表完整流程                                  |
|                                                                       |
|  服务A（发起方）                           服务B（接收方）                |
|       |                                         |                    |
|       |  +----------------------------------+   |                    |
|       |  |    本地事务（原子性保证）            |   |                    |
|       |  |  1. 执行业务操作（如创建订单）       |   |                    |
|       |  |  2. 向本地消息表插入消息记录         |   |                    |
|       |  |     status = PENDING              |   |                    |
|       |  +----------------------------------+   |                    |
|       |                                         |                    |
|       |  +------------------------------+       |                    |
|       |  |  消息扫描任务（定时任务）        |       |                    |
|       |  |  扫描 status=PENDING 的消息   |       |                    |
|       |  |  发送消息到 MQ               |       |                    |
|       |  +------------------------------+       |                    |
|       |              |                          |                    |
|       |              |  发送消息                 |                    |
|       |              +-------- MQ ------------->|                    |
|       |                                         |                    |
|       |                              +------------------------------+|
|       |                              |  服务B 消费消息（幂等处理）    ||
|       |                              |  执行本地业务（如扣减库存）    ||
|       |                              |  消费成功 -> 发送 ACK        ||
|       |                              +------------------------------+|
|       |                                         |                    |
|       |  收到 ACK 后：更新消息表 status = DONE   |                    |
|                                                                       |
|  如果消息发送失败：定时任务会重试，直到服务B成功消费                        |
|  如果服务B处理失败：服务B重新消费消息（需要幂等处理）                        |
+-----------------------------------------------------------------------+
```

### 2.3.3 本地消息表的优缺点

**优点：**
- 实现简单，不依赖特定框架
- 保证最终一致性
- 消息不会丢失（存在本地数据库）

**缺点：**
- 消息表与业务表耦合，增加业务数据库压力
- 需要额外的定时扫描任务
- 实时性较差（取决于定时任务间隔）
- 接收方需要做幂等处理

## 2.4 MQ 事务消息（RocketMQ）

### 2.4.1 Half 消息机制

RocketMQ 的事务消息通过**半事务消息（Half Message）**机制解决了本地消息表的耦合问题。

```
+-----------------------------------------------------------------------+
|                   RocketMQ 事务消息完整流程                              |
|                                                                       |
|  生产者（服务A）          RocketMQ Broker          消费者（服务B）         |
|       |                       |                        |              |
|       |  1. 发送 Half 消息      |                        |              |
|       +---------------------->|                        |              |
|       |  (消息对消费者不可见)    |                        |              |
|       |<--- 发送成功响应 --------|                        |              |
|       |                       |                        |              |
|       |  2. 执行本地事务        |                        |              |
|       |  +-----------------+  |                        |              |
|       |  | 创建订单          |  |                        |              |
|       |  | 扣减余额          |  |                        |              |
|       |  +-----------------+  |                        |              |
|       |                       |                        |              |
|       |  3. 提交/回滚事务消息   |                        |              |
|       +---- COMMIT ---------->|                        |              |
|       |                       |                        |              |
|       |                       |  4. 消息变为可见         |              |
|       |                       +----------------------->|              |
|       |                       |                        |              |
|       |                       |            5. 消费消息  |              |
|       |                       |            执行业务逻辑  |              |
|       |                       |<------ ACK ------------|              |
|                                                                       |
|  ======= 事务回查机制（生产者异常时触发）=======                           |
|       |<--- 事务回查 ---------|                        |              |
|       |  生产者检查本地事务状态  |                        |              |
|       |  -> 已提交：返回COMMIT  |                        |              |
|       |  -> 已回滚：返回ROLLBACK|                        |              |
|       +---- COMMIT/ROLLBACK -->|                        |              |
+-----------------------------------------------------------------------+
```

### 2.4.2 RocketMQ 事务消息代码示例

```java
// 本地事务监听器
@Component
public class OrderTransactionListener implements TransactionListener {

    @Autowired
    private OrderService orderService;

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 执行本地事务（Half消息发送成功后回调）
     */
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        OrderDTO orderDTO = (OrderDTO) arg;
        try {
            orderService.createOrder(orderDTO);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    /**
     * 事务回查（Broker 定时调用，检查本地事务状态）
     */
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        String orderId = new String(msg.getBody());
        Order order = orderMapper.selectById(orderId);
        if (order != null && order.getStatus() == OrderStatus.CREATED) {
            return LocalTransactionState.COMMIT_MESSAGE;
        }
        return LocalTransactionState.ROLLBACK_MESSAGE;
    }
}
```

# Part 3: Seata 整体架构

## 3.1 Seata 是什么

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的**分布式事务解决方案**，致力于在微服务架构下提供高性能和简单易用的分布式事务服务。

**发展历史：**

```
+-----------------------------------------------------------------+
|                       Seata 发展历程                              |
|                                                                 |
|  2014年 -- 阿里内部 TXC（Taobao Transaction Creator）诞生          |
|  2016年 -- GTS（Global Transaction Service）在阿里云商业化          |
|  2019年 -- Fescar（Fast & EaSy Commit And Rollback）开源          |
|  2019年 -- 更名为 Seata，加入 Apache 孵化器                         |
|  2020年 -- Seata 1.0 正式发布                                     |
|  2023年 -- Seata 2.0 发布，性能大幅提升                             |
+-----------------------------------------------------------------+
```

**Seata 的核心特性：**

- 支持 AT、TCC、Saga、XA 四种分布式事务模式
- 对业务代码侵入性极低（AT 模式）
- 高性能，支持大规模并发场景
- 支持多种数据库：MySQL、Oracle、PostgreSQL、TiDB 等
- 支持多种注册中心：Nacos、Eureka、ZooKeeper、Consul 等

---

## 3.2 Seata 三大角色

Seata 由三个核心组件构成：

### 3.2.1 TC（Transaction Coordinator，事务协调者）

TC 是 Seata Server 的核心，负责维护全局事务和分支事务的状态，驱动全局事务的提交或回滚。

**职责：**
- 维护全局事务的状态（活跃/提交/回滚）
- 维护分支事务的注册与状态
- 驱动全局事务的提交和回滚
- 管理全局锁

### 3.2.2 TM（Transaction Manager，事务管理器）

TM 是事务的发起者，通常是业务代码中使用 `@GlobalTransactional` 注解的那个服务。

**职责：**
- 向 TC 发起全局事务的开启请求（获取全局事务 XID）
- 在所有分支事务完成后，向 TC 提交或回滚全局事务

### 3.2.3 RM（Resource Manager，资源管理器）

RM 管理分支事务处理的资源，与 TC 通信，注册分支事务、汇报分支事务状态，并驱动分支事务的提交或回滚。

**职责：**
- 向 TC 注册分支事务
- 汇报分支事务执行状态（成功/失败）
- 接收 TC 的提交/回滚指令，驱动分支事务执行

### 3.2.4 三大角色架构图

```

|                         Seata 整体架构图                                  |
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |                    Seata Server（TC）                                 ||

|  |                                                                      ||
|  |  +----------------------------------------------------------+        ||

|  |  +----------------------------------------------------------+        ||

|  +-----------------------+         +-------------------------------+    |

|  |  订单服务               |         |                               |    |

+-------------------------------------------------------------------------+
```

# Part 4: AT 模式深度解析（核心）

## 4.1 AT 模式原理概述

AT（Auto Transaction）模式是 Seata 最核心、最常用的模式。它基于**支持本地 ACID 事务的关系型数据库**，在两阶段提交协议的基础上进行了优化：

- **第一阶段**：业务 SQL 执行，同时生成 undo log（回滚日志），本地事务立即提交（释放本地锁）
- **第二阶段（提交）**：异步清理 undo log，性能极高
- **第二阶段（回滚）**：根据 undo log 生成反向 SQL，执行回滚

**AT 模式的核心创新**：第一阶段就提交本地事务，极大减少了锁持有时间，提高了并发性能。

```
+-------------------------------------------------------------------------+
|                          AT 模式两阶段流程                                 |
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |                      第一阶段（Phase 1）                               ||
|  |                                                                      ||
|  |  1. 解析业务SQL，构造 SELECT...FOR UPDATE 查询 before image            ||
|  |  2. 执行业务SQL（UPDATE/INSERT/DELETE）                                ||
|  |  3. 查询 after image（根据主键）                                        ||
|  |  4. 生成 undo log = {before image, after image}                      ||
|  |  5. 向 TC 注册分支事务，申请全局锁                                       ||
|  |  6. 本地事务提交（业务数据 + undo log 原子性提交）                        ||
|  |  7. 向 TC 汇报分支事务状态（成功）                                       ||
|  +----------------------------------------------------------------------+|
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |                 第二阶段 - 全局提交（Phase 2 Commit）                  ||
|  |                                                                      ||
|  |  TC 决定全局提交（所有分支都成功）                                       ||
|  |  RM 异步删除 undo log（不在主链路上，性能极高）                           ||
|  |  释放全局锁                                                          ||
|  |  （业务数据在第一阶段已经提交，这里只是清理工作）                           ||
|  +----------------------------------------------------------------------+|
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |                 第二阶段 - 全局回滚（Phase 2 Rollback）                ||
|  |                                                                      ||
|  |  TC 决定全局回滚（某个分支失败）                                        ||
|  |  RM 根据 undo log 中的 before image 生成反向 SQL                     ||
|  |  执行反向 SQL，恢复数据到事务前的状态                                    ||
|  |  删除 undo log，提交回滚事务，释放全局锁                                ||
|  +----------------------------------------------------------------------+|
+-------------------------------------------------------------------------+
```

---

## 4.2 第一阶段：业务 SQL 执行 + 生成 undolog

### 4.2.1 undo_log 表结构

在使用 AT 模式的每个数据库中，都需要创建 `undo_log` 表：

```sql
-- undo_log 表（每个参与 AT 模式的数据库都需要建此表）
CREATE TABLE `undo_log` (
  `branch_id`     BIGINT(20)   NOT NULL COMMENT '分支事务ID',
  `xid`           VARCHAR(100) NOT NULL COMMENT '全局事务ID',
  `context`       VARCHAR(128) NOT NULL COMMENT '序列化类型，如jackson',
  `rollback_info` LONGBLOB     NOT NULL COMMENT '回滚日志（JSON，含before/after image）',
  `log_status`    INT(11)      NOT NULL COMMENT '状态：0=Normal, 1=GlobalFinished',
  `log_created`   DATETIME(6)  NOT NULL COMMENT '创建时间',
  `log_modified`  DATETIME(6)  NOT NULL COMMENT '修改时间',
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='AT事务模式的回滚日志表';
```

### 4.2.2 before image / after image 生成原理

Seata AT 模式通过代理 JDBC DataSource 来拦截 SQL 执行，在 SQL 执行前后分别查询数据快照。

**以 UPDATE 操作为例：**

业务 SQL：`UPDATE account SET balance = balance - 100 WHERE user_id = 'U001'`

**Before Image 生成过程：**

```
+-----------------------------------------------------------------------+
|              Before Image 生成（执行业务SQL之前）                         |
|                                                                       |
|  1. 解析 UPDATE SQL，提取 WHERE 条件：user_id = 'U001'                 |
|                                                                       |
|  2. 构造查询语句（加 FOR UPDATE 确保读到最新数据）：                       |
|     SELECT * FROM account WHERE user_id = 'U001' FOR UPDATE          |
|                                                                       |
|  3. 执行查询，得到 Before Image：                                       |
|     +-----------+-----------+----------+                              |
|     |  user_id  |  balance  |  status  |                              |
|     +-----------+-----------+----------+                              |
|     |   U001    |   1000    |    1     |                              |
|     +-----------+-----------+----------+                              |
+-----------------------------------------------------------------------+
```

**执行业务 SQL，After Image 查询：**

```
+-----------------------------------------------------------------------+
|              After Image 生成（执行业务SQL之后）                          |
|                                                                       |
|  执行：UPDATE account SET balance = balance - 100 WHERE user_id = 'U001'
|  结果：balance 从 1000 改为 900                                        |
|                                                                       |
|  根据 Before Image 中的主键值查询：                                      |
|     SELECT * FROM account WHERE user_id = 'U001'                     |
|                                                                       |
|  得到 After Image：                                                    |
|     +-----------+-----------+----------+                              |
|     |  user_id  |  balance  |  status  |                              |
|     +-----------+-----------+----------+                              |
|     |   U001    |    900    |    1     |                              |
|     +-----------+-----------+----------+                              |
+-----------------------------------------------------------------------+
```

**生成的 undo_log rollback_info（JSON 格式）：**

```json
{
  "sqlType": "UPDATE",
  "tableName": "account",
  "beforeImage": {
    "rows": [{
      "fields": [
        {"name": "user_id", "type": "VARCHAR", "value": "U001"},
        {"name": "balance", "type": "DECIMAL", "value": 1000},
        {"name": "status",  "type": "INTEGER", "value": 1}
      ]
    }]
  },
  "afterImage": {
    "rows": [{
      "fields": [
        {"name": "user_id", "type": "VARCHAR", "value": "U001"},
        {"name": "balance", "type": "DECIMAL", "value": 900},
        {"name": "status",  "type": "INTEGER", "value": 1}
      ]
    }]
  }
}
```

### 4.2.3 行锁（本地锁）持有时间对比

```
AT 模式本地锁持有时间（极短！）：
---------------------------------------------------------
开始 | SELECT FOR UPDATE | 执行SQL | 插入undo_log | 提交 | 结束
      ^                                               ^
  获取本地行锁                                     释放本地行锁

本地锁持有时间 = SQL执行时间 + undo_log插入时间（毫秒级）

传统 2PC 本地锁持有时间（很长！）：
---------------------------------------------------------
开始 | Prepare（锁定）| 等待协调者响应 | Commit | 结束
      ^                                        ^
  获取本地行锁                             释放本地行锁

2PC 锁持有时间包括等待协调者的网络往返时间，远大于AT模式！
```

---

## 4.3 第二阶段 Commit：异步清理 undolog

当 TM 决定全局提交时，TC 通知所有 RM 执行分支提交。由于第一阶段已经提交了本地事务，第二阶段只需**异步删除 undo log**，非常高效。

```java
// Seata RM 分支提交处理（简化版伪代码）
public BranchStatus branchCommit(BranchType branchType, String xid,
                                  long branchId, String resourceId,
                                  String applicationData) {
    // 异步删除 undo log，提交就是这么简单！
    asyncWorker.submit(() -> {
        // DELETE FROM undo_log WHERE xid=? AND branch_id=?
        undoLogManager.deleteUndoLog(xid, branchId, conn);
    });
    return BranchStatus.PhaseTwo_Committed;
}
```

**Phase2 Commit 时序图：**

```
+-----------------------------------------------------------------------+
|              AT 模式 Phase2 Commit 时序                                  |
|                                                                       |
|  TC                     RM_A                   RM_B                  |
|   |                       |                      |                   |
|   | 全局事务提交，通知各RM   |                      |                   |
|   |---branchCommit(A)---->|                      |                   |
|   |                       | 异步删除undo_log      |                   |
|   |                       | （不阻塞主链路）       |                   |
|   |<---ACK--------------- |                      |                   |
|   |---branchCommit(B)------------------------------>|               |
|   |                       |                      | 异步删除undo_log  |
|   |<---ACK-------------------------------------------------          |
|   |                       |                      |                   |
|   | 全局事务完成！           |                      |                   |
+-----------------------------------------------------------------------+
```

## 4.4 第二阶段 Rollback：根据 undolog 生成反向 SQL

当 TM 决定全局回滚时，TC 通知所有 RM 执行分支回滚。

```
+-----------------------------------------------------------------------+
|                    AT 模式回滚过程详解                                    |
|                                                                       |
|  已保存的 undo_log：                                                    |
|  before_image: {user_id=U001, balance=1000, status=1}                |
|  after_image:  {user_id=U001, balance=900,  status=1}                |
|                                                                       |
|  回滚步骤：                                                             |
|                                                                       |
|  Step 1: 查询当前数据（current image）：                                 |
|     SELECT * FROM account WHERE user_id = 'U001'                     |
|     -> current: {balance=900}                                        |
|                                                                       |
|  Step 2: 脏数据校验 - 比较 after_image 和 current image                 |
|     after_image.balance = 900                                         |
|     current.balance     = 900                                         |
|     -> 一致！可以安全回滚（数据未被其他事务修改）                           |
|                                                                       |
|  Step 3: 生成反向 SQL（用 before_image 覆盖）：                          |
|     UPDATE account SET balance = 1000 WHERE user_id = 'U001'         |
|                                                                       |
|  Step 4: 执行反向 SQL，恢复数据 -> balance = 1000                       |
|                                                                       |
|  Step 5: 删除 undo_log                                                |
|                                                                       |
|  Step 6: 提交本次回滚事务，释放全局锁                                    |
+-----------------------------------------------------------------------+
```

**脏数据校验失败场景（after_image != current）：**

```
+-----------------------------------------------------------------------+
|              脏数据校验失败 - 需要人工介入                                 |
|                                                                       |
|  场景：Seata 事务之外的代码（绕过DataSourceProxy）修改了数据                 |
|                                                                       |
|  after_image.balance = 900   (Seata事务执行后的值)                      |
|  current.balance     = 800   (被其他普通事务改为800了)                    |
|                                                                       |
|  -> 不一致！脏数据检查失败！                                             |
|  -> 抛出异常：DirtyDataConflictException                               |
|  -> 记录异常日志，发送告警                                               |
|  -> 需要 DBA 人工介入恢复数据                                            |
+-----------------------------------------------------------------------+
```

## 4.5 全局锁（Global Lock）机制详解

### 4.5.1 全局锁的作用

全局锁是 Seata AT 模式防止脏写的核心机制。在分布式事务环境下，单靠数据库本地锁无法防止跨服务的并发写入问题。

**无全局锁的问题场景：**

```
时间轴：

事务1（Seata全局事务）：
  T1: 读取 balance=1000，记录 before_image=1000
  T2: 执行 UPDATE balance=900
  T3: 本地事务提交，余额=900
  T5: 全局回滚，执行反向SQL，UPDATE balance=1000

事务2（普通本地事务，在T3-T5之间执行）：
      T4: 读取 balance=900
      T4: 执行 UPDATE balance=800
      T4: 本地事务提交

最终结果：balance=1000（事务2的修改被回滚操作覆盖！脏写！）
```

### 4.5.2 全局锁写隔离原理

```
+-----------------------------------------------------------------------+
|                       全局锁写隔离流程                                   |
|                                                                       |
|  Seata事务1（全局事务）           Seata事务2（全局事务）                   |
|        |                                  |                          |
|        | Step1: 向TC申请全局锁              |                          |
|        |    锁定：account表，主键=U001      |                          |
|        |<-- 全局锁获取成功                  |                          |
|        |                                  |                          |
|        | Step2: 提交本地事务（释放本地行锁）  |                          |
|        |    注意：全局锁仍然持有！            |                          |
|        |                                  |                          |
|        |                                  | Step3: 尝试申请全局锁       |
|        |                                  |    锁定：account:U001      |
|        |                                  |    全局锁冲突！             |
|        |                                  |    等待/重试（默认10次）...  |
|        |                                  |                          |
|        | Step4: 全局事务完成（提交或回滚）    |                          |
|        | Step5: 释放全局锁                  |                          |
|        |                                  |                          |
|        |                                  | Step6: 申请全局锁成功       |
|        |                                  |    继续执行业务              |
|                                                                       |
|  结论：全局锁保证了在Seata事务期间，其他Seata事务无法写入相同的数据行          |
+-----------------------------------------------------------------------+
```

**全局锁存储在 TC 的 lock_table 中：**

```sql
-- Seata TC Server 的 lock_table（存储全局锁信息）
CREATE TABLE `lock_table` (
  `row_key`        VARCHAR(128) NOT NULL COMMENT '行锁标识：resourceId^^^tableName^^^pk',
  `xid`            VARCHAR(128) DEFAULT NULL COMMENT '持有该锁的全局事务XID',
  `transaction_id` BIGINT(20)   DEFAULT NULL COMMENT '全局事务数字ID',
  `branch_id`      BIGINT(20)   NOT NULL COMMENT '分支事务ID',
  `resource_id`    VARCHAR(256) DEFAULT NULL COMMENT '资源ID（数据源标识）',
  `table_name`     VARCHAR(32)  DEFAULT NULL COMMENT '表名',
  `pk`             VARCHAR(36)  DEFAULT NULL COMMENT '主键值',
  `status`         TINYINT(4)   NOT NULL DEFAULT 0 COMMENT '0=locked, 1=rollbacking',
  `gmt_create`     DATETIME     DEFAULT NULL,
  `gmt_modified`   DATETIME     DEFAULT NULL,
  PRIMARY KEY (`row_key`),
  KEY `idx_status` (`status`),
  KEY `idx_branch_id` (`branch_id`),
  KEY `idx_xid` (`xid`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.5.3 读隔离（SELECT FOR UPDATE）

AT 模式默认的隔离级别是**读未提交（Read Uncommitted）**，即一个 Seata 事务可以读取到另一个 Seata 事务第一阶段已提交但全局未提交的数据（脏读）。

如果要求**读已提交**，需要在查询时使用 `SELECT FOR UPDATE`：

```java
// Mapper 接口
@Mapper
public interface AccountMapper {

    // 普通查询 - 可能脏读
    @Select("SELECT * FROM t_account WHERE user_id = #{userId}")
    Account selectByUserId(String userId);

    // SELECT FOR UPDATE - 会检查全局锁，实现读已提交
    @Select("SELECT * FROM t_account WHERE user_id = #{userId} FOR UPDATE")
    Account selectForUpdate(String userId);
}

// Service 使用
@GlobalTransactional
public void processAccount(String userId) {
    // 使用 FOR UPDATE 确保读取到全局事务提交后的一致数据
    Account account = accountMapper.selectForUpdate(userId);
    // ...
}
```

## 4.6 AT 模式脏读/脏写问题分析

### 4.6.1 脏写问题

**场景：Seata 事务 vs 非 Seata 事务（绕过全局锁的写操作）**

```
+-----------------------------------------------------------------------+
|              脏写问题分析                                                 |
|                                                                       |
|  问题场景：                                                             |
|    Seata事务1：修改 balance 1000->900，持有全局锁                        |
|    普通JDBC事务（非Seata）：直接 UPDATE balance=800，不检查全局锁          |
|    Seata事务1 回滚：执行反向SQL，balance->1000                           |
|                                                                       |
|    最终结果：balance=1000，普通事务的修改丢失！（脏写）                    |
|                                                                       |
|  解决方案：                                                             |
|    1. 所有写操作都必须经过 Seata 代理（DataSourceProxy）                  |
|    2. 不能绕过 Seata 直接操作数据库                                      |
|    3. 批量数据处理等场景需要特殊处理，避免绕过代理                          |
+-----------------------------------------------------------------------+
```

### 4.6.2 脏读问题

```
+-----------------------------------------------------------------------+
|              脏读问题分析                                                 |
|                                                                       |
|  AT模式默认允许脏读（读未提交）：                                          |
|                                                                       |
|  Seata事务1：修改 balance 1000->900，Phase1提交，等待Phase2              |
|  Seata事务2：SELECT balance，读到 900                                   |
|             （Phase2可能是提交也可能是回滚，此时读到的是"脏数据"）          |
|                                                                       |
|  如果Seata事务1最终回滚：                                                |
|    balance恢复为1000                                                   |
|    但Seata事务2基于900已做出错误决策（如判断余额足够扣款）                   |
|                                                                       |
|  解决方案：                                                             |
|    1. 对读一致性要求高的场景，使用 SELECT ... FOR UPDATE                  |
|    2. 使用 TCC 模式替代 AT 模式（TCC 有更强的隔离性）                     |
|    3. 业务层面做幂等处理和补偿                                            |
+-----------------------------------------------------------------------+
```

## 4.7 AT 模式性能分析

```
+-----------------------------------------------------------------------+
|                      AT 模式性能开销分析                                 |
|                                                                       |
|  Phase 1 额外开销（同步，在主链路上）：                                    |
|  1. SELECT before image：1次额外DB查询（小表单行查询，很快）               |
|  2. 执行业务SQL：正常开销                                                |
|  3. SELECT after image：1次额外DB查询                                  |
|  4. INSERT undo_log：1次额外DB写入（LONGBLOB，有一定开销）                |
|  5. 向TC申请全局锁：1次网络RPC（通常 < 1ms）                             |
|  6. 向TC注册分支事务：1次网络RPC                                         |
|  7. 向TC汇报分支状态：1次网络RPC                                         |
|                                                                       |
|  Phase 2 Commit 额外开销（异步，不在主链路上，不影响性能）：                 |
|  1. DELETE undo_log：1次异步DB删除                                     |
|  2. 向TC汇报提交状态：1次网络RPC                                         |
|                                                                       |
|  性能评估：                                                             |
|  本地事务：约 100%（基准）                                               |
|  AT 模式：约 150%~300%（额外2次DB查询 + 1次DB写入 + 3次RPC）              |
|                                                                       |
|  但是：AT 模式第一阶段立即释放锁，极大减少了锁冲突！                         |
|  高并发场景下，整体吞吐量可能远优于传统 2PC（2PC 持锁时间极长）               |
+-----------------------------------------------------------------------+
```

```java
// OrderServiceImpl.java（TM 发起方）
@Service
@Slf4j
public class OrderServiceImpl implements OrderService {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private StockFeignClient stockFeignClient;

    @Autowired
    private AccountFeignClient accountFeignClient;

    /**
     * 创建订单 - 全局事务入口
     * @GlobalTransactional 标注该方法为全局事务发起方（TM）
     * name: 全局事务名称，用于监控和日志
     * rollbackFor: 哪些异常触发回滚（建议用 Exception.class）
     * timeoutMills: 全局事务超时时间（毫秒），默认60000ms
     */
    @Override
    @GlobalTransactional(
        name = "seata-order-create",
        rollbackFor = Exception.class,
        timeoutMills = 30000
    )
    public void createOrder(String userId, String productId, Integer count) {
        log.info("====== 开始全局事务 ======");
        log.info("XID: {}", RootContext.getXID());
        log.info("创建订单，userId={}, productId={}, count={}", userId, productId, count);

        // Step 1: 创建订单（当前服务 DB 操作，自动成为分支事务 RM）
        Order order = new Order();
        order.setUserId(userId);
        order.setProductId(productId);
        order.setCount(count);
        order.setMoney(new BigDecimal(count).multiply(new BigDecimal("10")));
        order.setStatus(0);  // 创建中
        orderMapper.insert(order);
        log.info("Step1: 订单创建成功，orderId={}", order.getId());

        // Step 2: 扣减库存（调用库存服务 RM）
        // XID 会通过 Feign 拦截器自动传递到 HTTP Header: TX_XID
        log.info("Step2: 调用库存服务扣减库存，productId={}, count={}", productId, count);
        CommonResult stockResult = stockFeignClient.deductStock(productId, count);
        if (stockResult == null || stockResult.getCode() != 200) {
            throw new RuntimeException("扣减库存失败！" + (stockResult != null ? stockResult.getMessage() : "网络错误"));
        }

        // Step 3: 扣减余额（调用账户服务 RM）
        log.info("Step3: 调用账户服务扣减余额，userId={}, money={}", userId, order.getMoney());
        CommonResult accountResult = accountFeignClient.deductAccount(userId, order.getMoney());
        if (accountResult == null || accountResult.getCode() != 200) {
            throw new RuntimeException("扣减余额失败！" + (accountResult != null ? accountResult.getMessage() : "网络错误"));
        }

        // Step 4: 更新订单状态为已完成
        order.setStatus(1);
        orderMapper.updateById(order);
        log.info("Step4: 订单完成，orderId={}", order.getId());

        log.info("====== 全局事务提交 ======");
    }
}
```

# Part 5: TCC 模式深度解析

## 5.1 TCC 原理

TCC（Try-Confirm-Cancel）模式是一种应用层面的分布式事务解决方案，不依赖数据库的事务机制，通过在业务代码中实现三个阶段来保证分布式事务的一致性。

```
+-------------------------------------------------------------------------+
|                          TCC 三阶段详解                                   |
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |  Try 阶段（资源预留）                                                  ||
|  |                                                                      ||
|  |  完成所有业务检查，预留必要的业务资源。                                    ||
|  |  Try 成功后，资源被"冻结"，后续的 Confirm 或 Cancel 都能执行。            ||
|  |                                                                      ||
|  |  举例：转账100元，冻结转出方100元                                        ||
|  |  操作：balance - 100，frozen + 100（余额减少，冻结增加）                  ||
|  +----------------------------------------------------------------------+|
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |  Confirm 阶段（确认提交）                                               ||
|  |                                                                      ||
|  |  所有 Try 都成功后，执行真正的业务操作，使用 Try 阶段预留的资源。            ||
|  |  Confirm 操作必须是幂等的（因为 TC 可能会重试）。                         ||
|  |                                                                      ||
|  |  举例：将冻结的100元正式转出                                             ||
|  |  操作：frozen - 100（冻结释放，完成转账）                                ||
|  +----------------------------------------------------------------------+|
|                                                                         |
|  +----------------------------------------------------------------------+|
|  |  Cancel 阶段（取消释放）                                                ||
|  |                                                                      ||
|  |  Try 失败或发生异常时，释放 Try 阶段预留的资源。                           ||
|  |  Cancel 操作必须是幂等的（因为 TC 可能会重试）。                          ||
|  |                                                                      ||
|  |  举例：解冻之前冻结的100元                                               ||
|  |  操作：frozen - 100，balance + 100（冻结释放，余额恢复）                 ||
|  +----------------------------------------------------------------------+|
+-------------------------------------------------------------------------+
```

## 5.2 TCC 完整流程图

```

|                         TCC 完整流程图                                    |
|                                                                         |
|  正常流程（Try全部成功 -> Confirm）：                                       |
|                                                                         |
|  TM         TC          服务A（RM）        服务B（RM）                    |
|   |          |               |                  |                      |
|   |--开启全局事务-->|           |                  |                      |
|   |<--XID---------|           |                  |                      |

|   |<-Try(A)成功---|           |                   |                      |
|   |               |           |                   |                      |
|   |--Try(B)------------------------------------------->|                |
|   |               | <注册分支B----------------------|                    |
|   |               | ->注册成功------------------------->|               |
|   |               |           |                   | 执行Try：冻结资源    |
|   |<-Try(B)成功--- |           |                   |                    |
|   |               |           |                   |                    |
|   |--全局提交------>|           |                   |                    |
|   |               |--Confirm(A)-->|                |                    |
|   |               |           | 执行Confirm        |                    |

|   |<--提交完成-----|           |                   |                    |

|                                                                         |

|   |--全局回滚------>|               |                   |               |

|   |               |           | 执行Cancel：释放冻结    |               |
|   |               |<--ACK-----|                        |               |
|   |               |--Cancel(B)------------------------>|               |
|   |               |           |               执行Cancel               |
|   |               |<--ACK---------------------------------|             |
|   |<--回滚完成-----|           |                   |                    |
+-------------------------------------------------------------------------+
```

## 5.3 TCC 三个接口设计规范

```java
// TCC 接口定义规范
@LocalTCC
public interface AccountTccService {

    /**
     * Try 阶段：资源预留
     *
     * 设计规范：
     * 1. 必须进行参数校验和业务合法性检查
     * 2. 必须进行资源预留（冻结资源），不能执行真正的业务
     * 3. Try 结果决定后续是 Confirm 还是 Cancel
     * 4. 必须是幂等的（用 xid + action 做幂等）
     * 5. 必须处理悬挂（Cancel 先于 Try 执行的情况）
     *
     * @TwoPhaseBusinessAction：
     *   name:           分支事务名称（必须全局唯一）
     *   commitMethod:   Confirm 阶段方法名
     *   rollbackMethod: Cancel 阶段方法名
     */
    @TwoPhaseBusinessAction(
        name = "accountTccService",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryDeduct(
        BusinessActionContext actionContext,
        @BusinessActionContextParameter(paramName = "userId") String userId,
        @BusinessActionContextParameter(paramName = "amount") BigDecimal amount
    );

    /**
     * Confirm 阶段：确认提交
     *
     * 设计规范：
     * 1. 必须是幂等的（TC 可能会重试 Confirm）
     * 2. Confirm 理论上不能失败（失败会一直重试）
     * 3. 操作 Try 阶段预留的资源
     * 4. 参数只能从 BusinessActionContext 中获取
     */
    boolean confirm(BusinessActionContext actionContext);

    /**
     * Cancel 阶段：取消释放
     *
     * 设计规范：
     * 1. 必须是幂等的（TC 可能会重试 Cancel）
     * 2. 必须处理空回滚（Try 未执行时 Cancel 被调用）
     * 3. 必须处理悬挂（Cancel 先于 Try 执行）
     * 4. 参数只能从 BusinessActionContext 中获取
     */
    boolean cancel(BusinessActionContext actionContext);
}
```

### 5.4.2 TCC 实现类

```java
@Service
@Slf4j
public class AccountTccServiceImpl implements AccountTccService {

    @Autowired
    private AccountMapper accountMapper;

    @Autowired
    private TccActionLogMapper tccActionLogMapper;

    /**
     * Try 阶段：冻结资金（资源预留）
     */
    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean tryDeduct(BusinessActionContext actionContext,
                              String userId, BigDecimal amount) {
        String xid = actionContext.getXid();
        log.info("TCC Try, xid={}, userId={}, amount={}", xid, userId, amount);

        // === 1. 悬挂检查（防止 Cancel 先于 Try 执行后，Try 再来冻结资源）===
        TccActionLog cancelLog = tccActionLogMapper.selectByXidAndAction(xid, "cancel");
        if (cancelLog != null) {
            log.warn("【悬挂检查】Cancel 已执行，Try 拒绝执行！xid={}", xid);
            // 返回 false，Seata 会将分支事务标记为 PhaseOne_Failed
            return false;
        }

        // === 2. 幂等检查（Try 是否已执行过）===
        TccActionLog tryLog = tccActionLogMapper.selectByXidAndAction(xid, "try");
        if (tryLog != null) {
            log.info("【幂等检查】Try 已执行过，直接返回成功。xid={}", xid);
            return true;
        }

        // === 3. 参数校验 ===
        if (amount == null || amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("转账金额必须大于0，amount=" + amount);
        }

        // === 4. 业务合法性检查 ===
        Account account = accountMapper.selectByUserId(userId);
        if (account == null) {
            throw new RuntimeException("用户不存在: " + userId);
        }
        if (account.getResidue().compareTo(amount) < 0) {
            throw new RuntimeException("余额不足！可用余额=" + account.getResidue()
                + "，需要=" + amount);
        }

        // === 5. 资源预留（冻结资金）===
        // UPDATE t_account SET residue=residue-amount, frozen=frozen+amount
        // WHERE user_id=userId AND residue >= amount
        int rows = accountMapper.freeze(userId, amount);
        if (rows == 0) {
            throw new RuntimeException("冻结资金失败！可能是并发修改，userId=" + userId);
        }

        // === 6. 记录 Try 日志（用于幂等检查）===
        tccActionLogMapper.insert(TccActionLog.builder()
            .xid(xid).action("try").userId(userId).amount(amount).build());

        log.info("TCC Try 成功，xid={}, userId={}, 冻结金额={}", xid, userId, amount);
        return true;
    }

    /**
     * Confirm 阶段：扣减冻结资金（完成转账）
     */
    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean confirm(BusinessActionContext actionContext) {
        String xid = actionContext.getXid();
        String userId = (String) actionContext.getActionContext("userId");
        BigDecimal amount = new BigDecimal(
            actionContext.getActionContext("amount").toString());

        log.info("TCC Confirm, xid={}, userId={}, amount={}", xid, userId, amount);

        // 幂等检查
        TccActionLog confirmLog = tccActionLogMapper.selectByXidAndAction(xid, "confirm");
        if (confirmLog != null) {
            log.info("【幂等检查】Confirm 已执行过，直接返回成功。xid={}", xid);
            return true;
        }

        // 扣减冻结金额（frozen - amount）
        // UPDATE t_account SET frozen=frozen-amount WHERE user_id=userId AND frozen>=amount
        int rows = accountMapper.confirmDeduct(userId, amount);
        if (rows == 0) {
            throw new RuntimeException("Confirm 失败！冻结资金不足，userId=" + userId);
        }

        // 记录 Confirm 日志
        tccActionLogMapper.insert(TccActionLog.builder()
            .xid(xid).action("confirm").userId(userId).amount(amount).build());

        log.info("TCC Confirm 成功，xid={}", xid);
        return true;
    }

    /**
     * Cancel 阶段：释放冻结资金
     */
    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean cancel(BusinessActionContext actionContext) {
        String xid = actionContext.getXid();
        String userId = (String) actionContext.getActionContext("userId");
        BigDecimal amount = new BigDecimal(
            actionContext.getActionContext("amount").toString());

        log.info("TCC Cancel, xid={}, userId={}, amount={}", xid, userId, amount);

        // === 1. 空回滚处理 ===
        TccActionLog tryLog = tccActionLogMapper.selectByXidAndAction(xid, "try");
        if (tryLog == null) {
            log.warn("【空回滚】Try 未执行，无需释放资源，xid={}", xid);
            // 插入 cancel 记录，防止后续 Try 再来执行（悬挂处理）
            tccActionLogMapper.insert(TccActionLog.builder()
                .xid(xid).action("cancel").userId(userId).amount(amount).build());
            return true;
        }

        // === 2. 幂等检查 ===
        TccActionLog cancelLog = tccActionLogMapper.selectByXidAndAction(xid, "cancel");
        if (cancelLog != null) {
            log.info("【幂等检查】Cancel 已执行过，直接返回成功。xid={}", xid);
            return true;
        }

        // === 3. 释放冻结金额（frozen - amount，residue + amount）===
        // UPDATE t_account SET frozen=frozen-amount, residue=residue+amount
        // WHERE user_id=userId AND frozen>=amount
        int rows = accountMapper.cancelFreeze(userId, amount);
        if (rows == 0) {
            throw new RuntimeException("Cancel 失败！冻结资金不足，userId=" + userId);
        }

        // === 4. 记录 Cancel 日志 ===
        tccActionLogMapper.insert(TccActionLog.builder()
            .xid(xid).action("cancel").userId(userId).amount(amount).build());

        log.info("TCC Cancel 成功，xid={}", xid);
        return true;
    }
}
```

## 5.5 空回滚处理

**什么是空回滚？**

```
空回滚场景：

  TM 发起全局事务
  |
  调用服务A的 Try（网络超时，Try 未到达服务A）
  |
  TM 决定回滚（因为某个分支失败）
  |
  TC 发送 Cancel 到服务A
  |
  问题：服务A 没有执行过 Try，但收到了 Cancel！
       如果直接执行 Cancel 逻辑（释放冻结资源），会操作不存在的冻结数据，出错！

解决方案：Cancel 时检查 Try 是否执行，未执行则直接返回成功（空回滚）
```

---

## 5.6 悬挂处理

**什么是悬挂（Suspend）？**

```
悬挂场景（时序图）：

  TM       网络层         TC         服务A
   |         |              |            |
   |--Try(A)--+--网络堵塞--->X (Try请求在途) |
   |         |              |            |
   |  某分支失败              |            |
   |--回滚---->|              |            |
   |         |---Cancel(A)->|----------->|
   |         |              |       空回滚成功
   |         |              |       (记录cancel日志)
   |         |              |            |
   | 全局事务已结束            |            |
   |         |              |            |
   | 堵塞的Try请求终于到达...   |            |
   |         +-- 延迟到达 --->X----------->|
   |         |              |       检查cancel日志存在！
   |         |              |       -> 拒绝执行Try
   |         |              |       -> 防止资源悬挂
```

## 5.7 TCC vs AT 对比

```
+--------------+------------------------------+------------------------------+
|    维度       |         AT 模式               |         TCC 模式              |
+--------------+------------------------------+------------------------------+
|  代码侵入性   | 低，只需@GlobalTransactional  | 高，需实现Try/Confirm/Cancel   |
+--------------+------------------------------+------------------------------+
|  性能         | 中等（2次DB查询+1次DB写入）    | 高（无undo log，无全局行锁）      |
+--------------+------------------------------+------------------------------+
|  隔离性       | 弱（默认读未提交，脏读）         | 强（通过冻结字段自定义隔离）       |
+--------------+------------------------------+------------------------------+
|  数据库要求   | 需要 undo_log 表               | 需要冻结字段，无其他特殊要求       |
+--------------+------------------------------+------------------------------+
|  适用场景     | 标准CRUD，快速接入分布式事务     | 高并发、高性能、复杂业务逻辑       |
+--------------+------------------------------+------------------------------+
|  并发控制     | 全局锁（存在锁竞争）             | 数据库行级别的冻结字段（更轻量）    |
+--------------+------------------------------+------------------------------+
|  实现复杂度   | 简单，零业务代码改造             | 复杂（幂等/空回滚/悬挂三大挑战）   |
+--------------+------------------------------+------------------------------+
|  数据一致性   | 最终一致（中间可脏读）           | 较强（资源预留后不可见）           |
+--------------+------------------------------+------------------------------+
```

**选型建议：**
- 业务简单、快速接入：选 **AT 模式**
- 高并发、对性能要求高：选 **TCC 模式**
- 需要强隔离性：选 **TCC 模式**

# Part 6: Saga 模式深度解析

## 6.1 Saga 模式原理

Saga 模式将一个**长事务**拆分为一系列**短的本地事务**，每个本地事务都有对应的**补偿事务**。如果所有本地事务都成功，则整个 Saga 成功；如果某个本地事务失败，则逆序执行已成功事务的补偿，使数据恢复到初始状态。

```
+-------------------------------------------------------------------------+
|                           Saga 正向流程                                   |
|                                                                         |
|  T1          T2          T3          T4          T5                     |
|  创建订单 --> 扣减库存 --> 扣减余额 --> 通知快递 --> 发送邮件               |
|                                                                         |
|  每个 Ti 都是独立的本地事务，有对应的补偿事务 Ci：                            |
|                                                                         |
|  C1=取消订单  C2=恢复库存  C3=退款    C4=取消快递  C5=撤回邮件              |
|                                                                         |
|  ======================================================                  |
|                                                                         |
|                       Saga 补偿流程（T3 失败时）                           |
|                                                                         |
|  T1 成功    T2 成功    T3 失败！                                           |
|                           |                                             |
|                        触发补偿（逆序）                                    |
|                           |                                             |
|                       C2（恢复库存） <-- C1（取消订单）                      |
|                    逆序执行所有已成功事务的补偿                               |
+-------------------------------------------------------------------------+
```

**Saga 与其他模式的核心区别：**

```
2PC：所有参与者同时 Prepare，持锁等待，同时 Commit
     -> 强一致性，短事务，性能受锁限制

TCC：Try 阶段资源预留（冻结），Confirm 确认，Cancel 释放
     -> 较强隔离性，需要业务改造

Saga：本地事务依次执行，无全局锁，失败后补偿
     -> 最终一致性，中间状态可见，适合长事务
```

## 6.2 Saga 两种实现模式

### 6.2.1 编排模式（Orchestration）- 中央协调器

有一个中央协调器（Saga Orchestrator），负责调用各个服务并根据响应决定下一步操作。

```
+-------------------------------------------------------------------------+
|                       Saga 编排模式（Orchestration）                       |
|                                                                         |
|                   +---------------------+                               |
|                   |  Saga Orchestrator   |                               |
|                   |   （中央协调器）       |                               |
|                   |  状态机驱动执行流程    |                               |
|                   +---------------------+                               |
|                    |    |    |    |    |                                |
|              +-----+    |    |    |    +------+                         |
|              v          v    v    v           v                         |
|        +----------+  +---+  +---+  +---+  +----------+                |
|        | 服务A     |  |服 |  |服 |  |服 |  | 服务E     |                |
|        | 创建订单  |  |务 |  |务 |  |务 |  | 发送邮件  |                |
|        | 补偿:取消 |  |B  |  |C  |  |D  |  |          |                |
|        +----------+  +---+  +---+  +---+  +----------+                |
|                                                                         |
|  执行顺序（正常）：A -> B -> C -> D -> E（顺序执行）                        |
|  执行顺序（C失败）：C失败 -> 补偿B -> 补偿A（逆序补偿）                      |
|                                                                         |
|  优点：流程集中管理，易于监控和调试，状态清晰                                  |
|  缺点：Orchestrator 是单点，服务对 Orchestrator 有依赖                      |
+-------------------------------------------------------------------------+
```

### 6.2.2 协同模式（Choreography）- 事件驱动

没有中央协调器，每个服务完成本地事务后发布事件，下一个服务监听事件并执行。

```
+-------------------------------------------------------------------------+
|                       Saga 协同模式（Choreography）                        |
|                                                                         |
|  服务A          MQ           服务B          MQ          服务C            |
|   |              |               |              |              |         |
|   | T1:创建订单   |               |              |              |         |
|   +--发布事件---->|               |              |              |         |
|   |  "订单已创建" |<--订阅---------|              |              |         |
|   |              +--推送事件----->|              |              |         |
|   |              |  "订单已创建"  | T2:扣减库存   |              |         |
|   |              |              +--发布事件----->|              |         |
|   |              |               |  "库存已扣减" |<--订阅--------|         |
|   |              |               |              +--推送事件----->|         |
|   |              |               |              |  "库存已扣减" | T3:扣余额|
|   |              |               |              |              |         |
|  补偿流程（事件驱动的反向操作）：                                            |
|   T3失败 -> 发布"扣余额失败"事件 -> 服务B接收 -> 执行补偿 -> 发布补偿完成事件 |
|                                                                         |
|  优点：服务完全解耦，无单点，扩展性强                                         |
|  缺点：流程分散难以追踪，测试复杂，需要完善的事件溯源                           |
+-------------------------------------------------------------------------+
```

### 6.3.3 Saga 服务接口实现

```java
// 库存服务的 Saga 接口（正向 + 补偿）
@Service
@Slf4j
public class InventoryActionImpl implements InventoryAction {

    @Autowired
    private InventoryService inventoryService;

    /**
     * 正向方法：扣减库存
     * 方法名必须与状态机 JSON 中的 ServiceMethod 一致
     */
    @Override
    public boolean reduce(String businessKey, int count) {
        log.info("Saga 正向操作：扣减库存，businessKey={}, count={}", businessKey, count);
        try {
            inventoryService.reduce(businessKey, count);
            return true;
        } catch (Exception e) {
            log.error("库存扣减失败，businessKey={}", businessKey, e);
            return false;
        }
    }

    /**
     * 补偿方法：恢复库存
     *
     * 补偿方法设计要点：
     * 1. 必须是幂等的（可能被多次调用）
     * 2. 参数与正向方法相同或通过 businessKey 查询
     * 3. 补偿失败会一直重试，必须保证最终可执行成功
     */
    @Override
    public boolean compensateReduce(String businessKey) {
        log.info("Saga 补偿操作：恢复库存，businessKey={}", businessKey);
        try {
            // 查询原始扣减记录（通过businessKey）
            InventoryRecord record = inventoryService.findByBusinessKey(businessKey);
            if (record == null) {
                log.warn("未找到库存扣减记录，可能已补偿，businessKey={}", businessKey);
                return true;  // 幂等，直接返回成功
            }
            inventoryService.compensate(businessKey, record.getCount());
            return true;
        } catch (Exception e) {
            log.error("库存恢复失败，businessKey={}", businessKey, e);
            return false;
        }
    }
}

// 启动 Saga 状态机
@Service
@Slf4j
public class PurchaseService {

    @Autowired
    private StateMachineEngine stateMachineEngine;

    public void purchase(String userId, String productId, int count, BigDecimal amount) {
        Map<String, Object> startParams = new HashMap<>();
        String businessKey = UUID.randomUUID().toString();
        startParams.put("businessKey", businessKey);
        startParams.put("userId", userId);
        startParams.put("productId", productId);
        startParams.put("count", count);
        startParams.put("amount", amount);

        log.info("启动 Saga 状态机，businessKey={}", businessKey);

        // 同步启动（等待完成）
        StateMachineInstance inst = stateMachineEngine.start(
            "purchaseSaga",   // 状态机名称（与JSON的Name一致）
            null,             // tenantId，多租户场景使用
            startParams       // 启动参数
        );

        if (ExecutionStatus.SU.equals(inst.getStatus())) {
            log.info("Saga 执行成功！businessKey={}", businessKey);
        } else {
            log.error("Saga 执行失败！status={}, businessKey={}",
                inst.getStatus(), businessKey);
            throw new RuntimeException("购买失败：" + inst.getException());
        }
    }
}
```

## 6.4 Saga 使用场景

```
适合 Saga 模式的场景：

1. 长流程业务（执行时间跨越秒级甚至分钟级）
   示例：电商退款流程
         取消订单 -> 退货入库 -> 退还积分 -> 退款到账户 -> 通知物流
   原因：流程长，持锁成本极高，Saga 无锁执行

2. 遗留系统集成（无法改造的第三方服务）
   示例：需要调用银行接口、ERP 系统、外部支付系统
   原因：TCC 需要在服务内部增加 Try/Confirm/Cancel 接口
         AT 需要数据库支持 undo_log
         这些都无法对遗留系统改造

3. 跨企业的分布式事务
   示例：与合作方系统的数据交换
   原因：无法修改对方系统的数据库结构

4. 最终一致性要求的场景（非实时）
   示例：订单状态同步、数据归档、跨系统数据同步
```

## 6.5 Saga 优缺点

**优点：**
- 无全局锁，不会产生死锁，性能高
- 对遗留系统友好，只需增加补偿方法
- 每个步骤独立执行，单个步骤失败可以局部重试
- 没有长时间锁持有，适合跨数据库、跨系统场景

**缺点：**
- 中间状态可见（不满足隔离性），可能读到不一致数据
- 补偿逻辑复杂，需要业务支持可补偿（某些操作不可逆）
- 不支持强一致性，只有最终一致性
- 调试和排查故障比较困难

# Part 7: XA 模式详解

## 7.1 XA 协议原理

XA 协议是 X/Open 组织定义的分布式事务规范，是**数据库层面的两阶段提交标准**。几乎所有主流数据库（MySQL 5.7+、Oracle、PostgreSQL）都支持 XA 协议。

XA 协议的核心是定义了**事务管理器（TM）**和**资源管理器（RM）**之间的接口：

```
+-----------------------------------------------------------------------+
|                          XA 协议核心接口                                  |
|                                                                       |
|  xa_open()     - 初始化 RM                                             |
|  xa_close()    - 关闭 RM                                               |
|  xa_start()    - 开启分支事务，关联 XID                                 |
|  xa_end()      - 结束分支事务的输入阶段                                  |
|  xa_prepare()  - Phase 1：准备提交，写 undo/redo log，进入就绪状态         |
|  xa_commit()   - Phase 2：提交分支事务                                  |
|  xa_rollback() - Phase 2：回滚分支事务                                  |
|  xa_recover()  - 查询已 prepare 但未 commit/rollback 的事务（故障恢复）   |
+-----------------------------------------------------------------------+
```

## 7.2 Seata XA 模式实现

Seata XA 模式在 XA 协议之上做了一层封装，让 XA 模式也能统一纳入 Seata 的 TC-TM-RM 模型中管理。

```
+-----------------------------------------------------------------------+
|                      Seata XA 模式完整流程                                |
|                                                                       |
|  TM              TC（Seata Server）       RM_A（XA模式）    RM_B（XA模式）|
|   |                   |                      |                  |     |
|   | 1. 开启全局事务     |                      |                  |     |
|   +------------------>|                      |                  |     |
|   |<--XID-------------|                      |                  |     |
|   |                   |                      |                  |     |
|   | 2. 调用 RM_A       |                      |                  |     |
|   +---------------------------> XA START      |                  |     |
|   |                   |         执行业务SQL    |                  |     |
|   |                   |         XA END        |                  |     |
|   |                   | <注册分支事务(A)--------|                  |     |
|   |                   |                       |                  |     |
|   | 3. 调用 RM_B       |                       |                  |     |
|   +-------------------------------------------> XA START        |     |
|   |                   |                       |  执行业务SQL      |     |
|   |                   |                       |  XA END          |     |
|   |                   | <注册分支事务(B)--------|------------------|     |
|   |                   |                       |                  |     |
|   | 4. TM 决定提交     |                       |                  |     |
|   +------------------>|                       |                  |     |
|   |                   |--XA PREPARE(A)-------->|                  |     |
|   |                   |   XA PREPARE 完成      |                  |     |
|   |                   |<--OK------------------|                  |     |
|   |                   |--XA PREPARE(B)-------------------------->|     |
|   |                   |<--OK------------------------------------------  |
|   |                   |                       |                  |     |
|   |                   | (所有PREPARE成功，执行COMMIT)               |     |
|   |                   |--XA COMMIT(A)--------->|                  |     |
|   |                   |--XA COMMIT(B)-------------------------->  |     |
|   |<--提交完成---------|                       |                  |     |
|                                                                       |
|  如果 XA PREPARE 阶段某个 RM 失败：                                      |
|   |                   |--XA ROLLBACK(A)------->|                  |     |
|   |                   |--XA ROLLBACK(B)------------------------->  |     |
+-----------------------------------------------------------------------+
```

## 7.3 XA 模式配置

```java
// XA 模式数据源配置（Spring Boot）
@Configuration
public class DataSourceConfig {

    /**
     * 配置 XA 模式数据源
     * 使用 SeataDataSourceProxy 包装，并指定 BranchType.XA
     */
    @Bean
    @Primary
    @ConfigurationProperties(prefix = "spring.datasource")
    public DataSource dataSource(DataSourceProperties properties) {
        // 创建原始数据源（必须支持XA的驱动，如 MysqlXADataSource）
        MysqlXADataSource xaDataSource = new MysqlXADataSource();
        xaDataSource.setUrl(properties.getUrl());
        xaDataSource.setUser(properties.getUsername());
        xaDataSource.setPassword(properties.getPassword());

        // 用 Atomikos 包装 XA 数据源
        AtomikosDataSourceBean atomikosDataSource = new AtomikosDataSourceBean();
        atomikosDataSource.setXaDataSource(xaDataSource);
        atomikosDataSource.setUniqueResourceName("order-xa-datasource");

        // 用 SeataDataSourceProxy 包装，指定 BranchType.XA
        return new DataSourceProxyXA(atomikosDataSource);
    }
}
```

## 7.4 XA vs AT 对比

```
+--------------+------------------------------+------------------------------+
|    维度       |         AT 模式               |         XA 模式              |
+--------------+------------------------------+------------------------------+
|  一致性级别   | 弱（依赖全局锁+undo补偿）       | 强（数据库原生ACID保证）         |
+--------------+------------------------------+------------------------------+
|  性能         | 较好（Phase1提交，Phase2异步） | 差（Phase1持锁，同步等待）       |
+--------------+------------------------------+------------------------------+
|  数据库支持   | 所有支持JDBC的关系数据库        | 必须支持XA协议（MySQL5.7+）     |
+--------------+------------------------------+------------------------------+
|  侵入性       | 需要 undo_log 表               | 无需额外表，数据库原生支持        |
+--------------+------------------------------+------------------------------+
|  适用场景     | 绝大多数业务场景，推荐优先使用   | 需要强一致性的金融类核心业务       |
+--------------+------------------------------+------------------------------+
|  锁持有时间   | 极短（Phase1提交即释放）        | 较长（Phase1到Phase2全程持锁）   |
+--------------+------------------------------+------------------------------+
|  故障恢复     | 依赖 undo log 回放             | 数据库原生 xa_recover 机制       |
+--------------+------------------------------+------------------------------+
```

**XA 适用场景：**
- 对数据一致性要求极高的场景（如银行核心系统）
- 有监管合规要求，必须强一致性
- 并发不高但一致性要求 100% 保证的场景

### 8.3.1 TC Server 数据库建表脚本

```sql
-- ================================================================
-- Seata Server 使用的三张核心表（在 seata 数据库中创建）
-- ================================================================

CREATE DATABASE IF NOT EXISTS seata DEFAULT CHARACTER SET utf8mb4;
USE seata;

-- global_table: 全局事务表，记录每个全局事务的状态
CREATE TABLE IF NOT EXISTS `global_table` (
  `xid`                       VARCHAR(128) NOT NULL COMMENT '全局事务ID',
  `transaction_id`            BIGINT(20)   DEFAULT NULL COMMENT '全局事务数字ID',
  `status`                    TINYINT(4)   NOT NULL COMMENT '事务状态',
  `application_id`            VARCHAR(32)  DEFAULT NULL COMMENT '应用ID',
  `transaction_service_group` VARCHAR(32)  DEFAULT NULL COMMENT '服务事务组',
  `transaction_name`          VARCHAR(128) DEFAULT NULL COMMENT '事务名',
  `timeout`                   INT(11)      DEFAULT NULL COMMENT '事务超时时间（ms）',
  `begin_time`                BIGINT(20)   DEFAULT NULL COMMENT '事务开始时间',
  `application_data`          VARCHAR(2000) DEFAULT NULL COMMENT '应用数据',
  `gmt_create`                DATETIME     DEFAULT NULL COMMENT '创建时间',
  `gmt_modified`              DATETIME     DEFAULT NULL COMMENT '修改时间',
  PRIMARY KEY (`xid`),
  KEY `idx_status_gmt_modified` (`status`, `gmt_modified`),
  KEY `idx_transaction_id` (`transaction_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- branch_table: 分支事务表，记录每个分支事务的状态
CREATE TABLE IF NOT EXISTS `branch_table` (
  `branch_id`         BIGINT(20)    NOT NULL COMMENT '分支事务ID',
  `xid`               VARCHAR(128)  NOT NULL COMMENT '所属全局事务ID',
  `transaction_id`    BIGINT(20)    DEFAULT NULL COMMENT '全局事务数字ID',
  `resource_group_id` VARCHAR(32)   DEFAULT NULL COMMENT '资源组ID',
  `resource_id`       VARCHAR(256)  DEFAULT NULL COMMENT '资源ID（数据源URL）',
  `branch_type`       VARCHAR(8)    DEFAULT NULL COMMENT '分支类型：AT/TCC/SAGA/XA',
  `status`            TINYINT(4)    DEFAULT NULL COMMENT '分支事务状态',
  `client_id`         VARCHAR(64)   DEFAULT NULL COMMENT '客户端ID',
  `application_data`  VARCHAR(2000) DEFAULT NULL COMMENT '应用数据（如TCC参数）',
  `gmt_create`        DATETIME(6)   DEFAULT NULL,
  `gmt_modified`      DATETIME(6)   DEFAULT NULL,
  PRIMARY KEY (`branch_id`),
  KEY `idx_xid` (`xid`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- lock_table: 全局锁表，记录AT模式的全局锁信息
CREATE TABLE IF NOT EXISTS `lock_table` (
  `row_key`        VARCHAR(128) NOT NULL COMMENT '行锁标识',
  `xid`            VARCHAR(128) DEFAULT NULL COMMENT '持有该锁的全局事务XID',
  `transaction_id` BIGINT(20)   DEFAULT NULL,
  `branch_id`      BIGINT(20)   NOT NULL,
  `resource_id`    VARCHAR(256) DEFAULT NULL COMMENT '资源ID',
  `table_name`     VARCHAR(32)  DEFAULT NULL COMMENT '表名',
  `pk`             VARCHAR(36)  DEFAULT NULL COMMENT '主键值',
  `status`         TINYINT(4)   NOT NULL DEFAULT 0 COMMENT '0=locked, 1=rollbacking',
  `gmt_create`     DATETIME     DEFAULT NULL,
  `gmt_modified`   DATETIME     DEFAULT NULL,
  PRIMARY KEY (`row_key`),
  KEY `idx_status` (`status`),
  KEY `idx_branch_id` (`branch_id`),
  KEY `idx_xid_and_branch_id` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- distributed_lock: 分布式锁表（Seata 2.x 新增，TC集群选主用）
CREATE TABLE IF NOT EXISTS `distributed_lock` (
  `lock_key`   CHAR(20)     NOT NULL COMMENT '锁键',
  `lock_value` VARCHAR(520) NOT NULL COMMENT '锁值（持有者信息）',
  `expire`     BIGINT(20)   DEFAULT NULL COMMENT '过期时间（毫秒时间戳）',
  PRIMARY KEY (`lock_key`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 9.3 DataSourceProxy 配置（AT 模式数据源代理）

AT 模式的核心是通过 `DataSourceProxy` 拦截 JDBC 操作，自动生成 undo log 和注册分支事务。

```java
/**
 * Seata AT 模式数据源代理配置
 *
 * 重要：在 Spring Boot 中需要手动配置 DataSourceProxy，
 * 防止 Spring 使用原始 DataSource（绕过 Seata 代理）
 */
@Configuration
public class SeataDataSourceConfig {

    /**
     * 注入原始 DataSource（Druid/HikariCP 等）
     */
    @Bean
    @ConfigurationProperties(prefix = "spring.datasource")
    public DataSource druidDataSource() {
        return DruidDataSourceBuilder.create().build();
    }

    /**
     * 用 DataSourceProxy 包装原始 DataSource
     * Seata 通过代理拦截所有 SQL 操作：
     * 1. 在 SQL 执行前后生成 before/after image
     * 2. 生成并保存 undo log
     * 3. 向 TC 注册分支事务
     * 4. 申请全局锁
     *
     * 注意：@Primary 确保 Spring 使用这个代理数据源，而不是原始数据源
     */
    @Primary
    @Bean("dataSource")
    public DataSourceProxy dataSourceProxy(DataSource druidDataSource) {
        return new DataSourceProxy(druidDataSource);
    }

    /**
     * 使用代理数据源创建 SqlSessionFactory
     * 确保 MyBatis 使用代理数据源
     */
    @Bean
    public SqlSessionFactory sqlSessionFactoryBean(DataSourceProxy dataSourceProxy)
            throws Exception {
        SqlSessionFactoryBean sqlSessionFactoryBean = new SqlSessionFactoryBean();
        sqlSessionFactoryBean.setDataSource(dataSourceProxy);
        sqlSessionFactoryBean.setMapperLocations(
            new PathMatchingResourcePatternResolver()
                .getResources("classpath:mapper/*.xml")
        );
        return sqlSessionFactoryBean.getObject();
    }
}
```


```java
// 错误示例1：同一个类中调用，@GlobalTransactional 无效（AOP 代理问题）
@Service
public class OrderService {
    public void outer() {
        this.inner();  // 同类内部调用，绕过代理，@GlobalTransactional 无效！
    }

    @GlobalTransactional
    public void inner() { ... }
}

// 正确示例1：通过注入自身或独立 Service 调用

    @Autowired
    private OrderService self;  // 注入自身（Spring 代理对象）

    public void outer() {
        self.inner();  // 通过代理对象调用，@GlobalTransactional 生效
    }

    @GlobalTransactional
    public void inner() { ... }
}

// 错误示例2：@GlobalTransactional 标注在 private 方法上（AOP 代理无法拦截）
@GlobalTransactional  // 无效！
private void createOrder() { ... }

// 正确：只能标注在 public 方法上
@GlobalTransactional
public void createOrder() { ... }
```

## 9.5 Feign 调用中 XID 传递

在微服务调用中，全局事务 XID 必须通过 HTTP Header 传递到下游服务，否则下游服务无法加入全局事务。

```java
/**
 * Feign 拦截器：在请求头中传递 Seata XID
 * 在 Spring Cloud Alibaba 2022+ 中，这个拦截器已经自动配置
 * 但如果需要自定义，可以手动配置：
 */
@Configuration
public class FeignSeataConfig {

    @Bean
    public RequestInterceptor seataFeignRequestInterceptor() {
        return requestTemplate -> {
            // 从当前线程获取 XID
            String xid = RootContext.getXID();
            if (StringUtils.isNotBlank(xid)) {
                // 将 XID 放到 HTTP Header 中传递给下游服务
                requestTemplate.header(RootContext.KEY_XID, xid);
                log.debug("传递 Seata XID: {}", xid);
            }
        };
    }
}

/**
 * 下游服务接收 XID（通过 SeataHandlerInterceptor 自动处理）
 * Spring Cloud Alibaba 会自动注册拦截器，从 Header 中读取 XID 并绑定到线程
 *
 * 如果需要手动处理：
 */
@Component
public class SeataXIDInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                              HttpServletResponse response, Object handler) {
        // 从 Header 中读取 XID
        String xid = request.getHeader(RootContext.KEY_XID);
        if (StringUtils.isNotBlank(xid)) {
            // 绑定到当前线程（RM 会通过此 XID 注册到全局事务）
            RootContext.bind(xid);
            log.debug("接收到 Seata XID: {}", xid);
        }
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                 HttpServletResponse response,
                                 Object handler, Exception ex) {
        // 清理线程绑定的 XID
        if (StringUtils.isNotBlank(RootContext.getXID())) {
            RootContext.unbind();
        }
    }
}
```

## 10.2 案例2：TCC 转账（跨银行转账，资源预留模式）

### 10.2.1 业务场景说明

```
跨银行转账场景：
  用户 A（工商银行账户）向用户 B（建设银行账户）转账 1000 元

  服务架构：
  TransferService（TM）
    -> 工商银行服务（TCC RM）
    -> 建设银行服务（TCC RM）

  TCC 三阶段：
  Try：工商银行冻结 A 的 1000 元；建设银行为 B 预留 1000 元接收槽
  Confirm：工商银行扣减 A 的冻结金额；建设银行给 B 增加 1000 元
  Cancel：工商银行解冻 A 的 1000 元；建设银行释放 B 的预留槽
```

### 10.2.2 转账服务（TM）

```java
@Service
@Slf4j
public class TransferService {

    @Autowired
    private ICBCTccService icbcTccService;   // 工商银行 TCC 接口

    @Autowired
    private CCBTccService ccbTccService;     // 建设银行 TCC 接口

    /**
     * 跨行转账（全局事务发起方）
     */
    @GlobalTransactional(name = "cross-bank-transfer", rollbackFor = Exception.class)
    public void transfer(String fromUserId, String toUserId, BigDecimal amount) {
        log.info("开始跨行转账，from={}, to={}, amount={}", fromUserId, toUserId, amount);

        // Try 阶段：由 Seata 框架自动调用
        // 1. 调用工商银行 Try：冻结转出方资金
        boolean icbcTryResult = icbcTccService.tryDeduct(
            RootContext.getActionContext(),
            fromUserId,
            amount
        );
        if (!icbcTryResult) {
            throw new RuntimeException("工商银行账户余额不足或账户异常");
        }

        // 2. 调用建设银行 Try：预留接收槽
        boolean ccbTryResult = ccbTccService.tryReceive(
            RootContext.getActionContext(),
            toUserId,
            amount
        );
        if (!ccbTryResult) {
            throw new RuntimeException("建设银行账户不可用或预留失败");
        }

        log.info("转账 Try 阶段成功，等待 Confirm...");
        // Seata 自动执行 Confirm 或 Cancel
    }
}
```

### 10.2.3 工商银行 TCC 实现

```java
@LocalTCC
public interface ICBCTccService {

    @TwoPhaseBusinessAction(
        name = "icbcTccService",
        commitMethod = "confirmDeduct",
        rollbackMethod = "cancelDeduct"
    )
    boolean tryDeduct(
        BusinessActionContext actionContext,
        @BusinessActionContextParameter(paramName = "fromUserId") String fromUserId,
        @BusinessActionContextParameter(paramName = "amount") BigDecimal amount
    );

    boolean confirmDeduct(BusinessActionContext actionContext);

    boolean cancelDeduct(BusinessActionContext actionContext);
}

@Service
@Slf4j
public class ICBCTccServiceImpl implements ICBCTccService {

    @Autowired
    private ICBCAccountMapper accountMapper;

    @Autowired
    private TccLogMapper tccLogMapper;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean tryDeduct(BusinessActionContext ctx,
                              String fromUserId, BigDecimal amount) {
        String xid = ctx.getXid();

        // 悬挂检查
        if (tccLogMapper.existsByXidAndAction(xid, "cancel")) {
            return false;
        }
        // 幂等检查
        if (tccLogMapper.existsByXidAndAction(xid, "try")) {
            return true;
        }

        // 检查余额并冻结
        ICBCAccount account = accountMapper.selectForUpdate(fromUserId);
        if (account.getBalance().compareTo(amount) < 0) {
            throw new RuntimeException("余额不足");
        }

        accountMapper.freeze(fromUserId, amount);
        tccLogMapper.insert(xid, "try", fromUserId, amount);

        log.info("ICBC Try 成功，冻结 {} 元，xid={}", amount, xid);
        return true;
    }

    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean confirmDeduct(BusinessActionContext ctx) {
        String xid = ctx.getXid();
        String fromUserId = (String) ctx.getActionContext("fromUserId");
        BigDecimal amount = new BigDecimal(ctx.getActionContext("amount").toString());

        // 幂等检查
        if (tccLogMapper.existsByXidAndAction(xid, "confirm")) {
            return true;
        }

        // 扣减冻结金额（转账完成）
        accountMapper.confirmDeduct(fromUserId, amount);
        tccLogMapper.insert(xid, "confirm", fromUserId, amount);

        log.info("ICBC Confirm 成功，扣减 {} 元，xid={}", amount, xid);
        return true;
    }

    @Override
    @Transactional(rollbackFor = Exception.class)
    public boolean cancelDeduct(BusinessActionContext ctx) {
        String xid = ctx.getXid();
        String fromUserId = (String) ctx.getActionContext("fromUserId");
        BigDecimal amount = new BigDecimal(ctx.getActionContext("amount").toString());

        // 空回滚
        if (!tccLogMapper.existsByXidAndAction(xid, "try")) {
            tccLogMapper.insert(xid, "cancel", fromUserId, amount);
            return true;
        }
        // 幂等检查
        if (tccLogMapper.existsByXidAndAction(xid, "cancel")) {
            return true;
        }

        // 释放冻结
        accountMapper.unfreeze(fromUserId, amount);
        tccLogMapper.insert(xid, "cancel", fromUserId, amount);

        log.info("ICBC Cancel 成功，解冻 {} 元，xid={}", amount, xid);
        return true;
    }
}
```


## 11.1 @GlobalTransactional 注解处理（GlobalTransactionalInterceptor）

```java
/**
 * Seata 全局事务 AOP 拦截器
 * 位置：io.seata.spring.annotation.GlobalTransactionalInterceptor
 *
 * Spring 启动时，GlobalTransactionScanner 会扫描所有 @GlobalTransactional
 * 方法，并创建 AOP 代理，将方法调用委托给此拦截器处理
 */
public class GlobalTransactionalInterceptor implements MethodInterceptor {

    private final FailureHandler failureHandler;
    private final TransactionalTemplate transactionalTemplate;

    @Override
    public Object invoke(MethodInvocation methodInvocation) throws Throwable {
        // 1. 获取 @GlobalTransactional 注解配置
        GlobalTransactional globalTransactional =
            methodInvocation.getMethod().getAnnotation(GlobalTransactional.class);

        if (globalTransactional != null) {
            // 2. 进入全局事务逻辑
            return handleGlobalTransaction(methodInvocation, globalTransactional);
        }

        // 没有注解，直接执行原方法
        return methodInvocation.proceed();
    }

    private Object handleGlobalTransaction(MethodInvocation methodInvocation,
                                            GlobalTransactional globalTransactional)
            throws Throwable {
        // 通过 TransactionalTemplate 执行全局事务
        return transactionalTemplate.execute(new TransactionalExecutor() {
            @Override
            public Object execute() throws Throwable {
                // 执行原始业务方法
                return methodInvocation.proceed();
            }

            @Override
            public TransactionInfo getTransactionInfo() {
                return new TransactionInfo(
                    globalTransactional.timeoutMills(),
                    globalTransactional.name(),
                    globalTransactional.rollbackFor(),
                    globalTransactional.noRollbackFor(),
                    globalTransactional.propagation()
                );
            }
        });
    }
}
```

## 11.2 TM 发起全局事务流程（TransactionalTemplate）

```java
/**
 * 全局事务执行模板
 * 位置：io.seata.tm.api.TransactionalTemplate
 */
public class TransactionalTemplate {

    public Object execute(TransactionalExecutor business) throws Throwable {
        // 1. 获取或创建事务信息
        TransactionInfo txInfo = business.getTransactionInfo();

        // 2. 根据传播级别决定是否开启新事务
        GlobalTransaction tx = GlobalTransactionContext.getCurrentOrCreate();

        // 3. 开启全局事务（向 TC 发送 begin 请求）
        try {
            beginTransaction(txInfo, tx);
        } catch (TransactionException e) {
            assertRollback(e, txInfo);
        }

        Object rs;
        try {
            // 4. 执行业务代码（调用各 RM 服务）
            rs = business.execute();
        } catch (Throwable ex) {
            // 5. 业务异常，执行全局回滚
            completeTransactionAfterThrowing(txInfo, tx, ex);
            throw ex;
        }

        // 6. 业务成功，执行全局提交
        commitTransaction(tx);
        return rs;
    }

    private void beginTransaction(TransactionInfo txInfo, GlobalTransaction tx)
            throws TransactionException {
        // 向 TC 发送 GlobalBeginRequest
        // TC 生成 XID，存储到 global_table
        // XID 格式：{TC IP}:{TC Port}:{事务ID序列}
        // 示例：192.168.1.100:8091:2023050800000123
        tx.begin(txInfo.getTimeOut(), txInfo.getName());

        // 将 XID 绑定到当前线程
        RootContext.bind(tx.getXid());
    }

    private void commitTransaction(GlobalTransaction tx) throws TransactionException {
        // 向 TC 发送 GlobalCommitRequest
        // TC 将 global_table 中该事务状态改为 Committing
        // TC 异步通知所有 RM 执行 branchCommit
        tx.commit();
    }

    private void completeTransactionAfterThrowing(TransactionInfo txInfo,
                                                   GlobalTransaction tx,
                                                   Throwable originalException)
            throws TransactionException {
        // 判断是否需要回滚
        if (txInfo.rollbackOn(originalException)) {
            tx.rollback();  // 向 TC 发送 GlobalRollbackRequest
        } else {
            tx.commit();    // 某些异常不触发回滚
        }
    }
}
```

## 11.3 RM 分支事务注册流程

```java
/**
 * AT 模式 RM 分支事务注册流程（简化版）
 * 位置：io.seata.rm.datasource.exec.AbstractDMLBaseExecutor
 */
public abstract class AbstractDMLBaseExecutor<T, S extends Statement>
        extends BaseTransactionalExecutor<T, S> {

    @Override
    public T doExecute(Object... args) throws Throwable {
        AbstractConnectionProxy connectionProxy = statementProxy.getConnectionProxy();

        // 1. 查询 Before Image（执行业务SQL之前）
        TableRecords beforeImage = beforeImage();

        // 2. 执行业务 SQL
        T result = statementCallback.execute(statementProxy.getTargetStatement(), args);

        // 3. 查询 After Image（执行业务SQL之后）
        TableRecords afterImage = afterImage(beforeImage);

        // 4. 生成并持久化 undo log
        // (在当前本地事务中，与业务SQL同一个连接)
        prepareUndoLog(beforeImage, afterImage);

        return result;
    }
}

/**
 * AT 模式连接代理 - 在 commit 时注册分支事务和全局锁
 * 位置：io.seata.rm.datasource.ConnectionProxy
 */
public class ConnectionProxy extends AbstractConnectionProxy {

    @Override
    public void commit() throws SQLException {
        try {
            // 在同一个本地事务中，将 undo_log 和业务数据一起提交
            if (context.hasUndoLog() && context.inGlobalTransaction()) {
                // 5. 向 TC 注册分支事务（获取分支事务 branchId）
                register();
            }
        } catch (TransactionException e) {
            recognizeLockKeyConflictException(e, context.buildLockKeys());
        }
        // 6. 提交本地事务（业务数据 + undo_log 原子提交）
        targetConnection.commit();
        // 7. 向 TC 汇报分支事务状态
        report(true);
        // 8. 清理当前连接的上下文
        context.reset();
    }

    private void register() throws TransactionException {
        if (!context.hasLockKeys()) {
            return;
        }
        // 向 TC 发送 BranchRegisterRequest
        // 申请全局锁（lockKeys = tableName:pk1,pk2,...）
        Long branchId = DefaultResourceManager.get().branchRegister(
            BranchType.AT,
            getDataSourceProxy().getResourceId(),
            null,
            context.getXid(),
            null,
            context.buildLockKeys()  // 全局锁key
        );
        context.setBranchId(branchId);
    }
}
```

## 11.5 undolog 生成核心源码

```java
/**
 * UPDATE SQL 执行器
 * 位置：io.seata.rm.datasource.exec.UpdateExecutor
 */
public class UpdateExecutor<T, S extends Statement>
        extends AbstractDMLBaseExecutor<T, S> {

    /**
     * 生成 Before Image
     * 步骤：解析UPDATE的WHERE条件 -> 构造SELECT查询 -> 执行查询
     */
    @Override
    protected TableRecords beforeImage() throws SQLException {
        // 1. 解析 UPDATE SQL，获取涉及的表名和 WHERE 条件
        SQLUpdateRecognizer sqlUpdateRecognizer = (SQLUpdateRecognizer) sqlRecognizer;
        TableMeta tableMeta = getTableMeta(sqlUpdateRecognizer.getTableName());

        // 2. 构造 SELECT ... FOR UPDATE 查询
        // 目的：获取将被更新的行的当前数据
        // FOR UPDATE 确保读到最新数据（避免并发问题）
        String selectSQL = buildBeforeImageSQL(tableMeta, sqlUpdateRecognizer);

        // 3. 执行查询，得到 before image
        return buildTableRecords(tableMeta, selectSQL);
    }

    /**
     * 生成 After Image
     * 步骤：根据 before image 的主键值重新查询
     */
    @Override
    protected TableRecords afterImage(TableRecords beforeImage) throws SQLException {
        if (beforeImage == null || beforeImage.isEmpty()) {
            return TableRecords.empty(getTableMeta(sqlRecognizer.getTableName()));
        }

        // 根据 before image 的主键构造查询
        String afterImageSQL = buildAfterImageSQL(
            getTableMeta(sqlRecognizer.getTableName()), beforeImage);

        return buildTableRecords(getTableMeta(sqlRecognizer.getTableName()), afterImageSQL);
    }
}

/**
 * UndoLog 管理器
 * 位置：io.seata.rm.datasource.undo.AbstractUndoLogManager
 */
public abstract class AbstractUndoLogManager implements UndoLogManager {

    /**
     * 插入 undo log 到数据库
     * 在 commit 时与业务数据同一个本地事务提交
     */
    @Override
    public void flushUndoLogs(ConnectionProxy cp) throws SQLException {
        ConnectionContext connectionContext = cp.getContext();

        // 序列化 undo log 为 JSON
        BranchUndoLog branchUndoLog = new BranchUndoLog();
        branchUndoLog.setXid(connectionContext.getXid());
        branchUndoLog.setBranchId(connectionContext.getBranchId());
        branchUndoLog.setSqlUndoLogs(connectionContext.getUndoItems());

        UndoLogParser parser = UndoLogParserFactory.getInstance();
        byte[] undoLogContent = parser.encode(branchUndoLog);

        // INSERT INTO undo_log (branch_id, xid, context, rollback_info, log_status, log_created, log_modified)
        // VALUES (?, ?, ?, ?, ?, now(6), now(6))
        insertUndoLogWithNormal(
            connectionContext.getXid(),
            connectionContext.getBranchId(),
            buildContext(parser.getName()),
            undoLogContent,
            cp.getTargetConnection()
        );
    }
}
```

## 13.2 网络分区导致的脑裂

### 问题描述

```
场景：
  Seata Server 集群中，节点间发生网络分区
  部分节点认为自己是 Leader，多个节点同时处理事务

表现：
  同一个 XID 被多个 TC 节点处理
  全局事务状态不一致
  部分分支提交，部分分支回滚
```

### 解决方案

```
1. Seata 2.x 使用 Raft 协议解决脑裂问题：
   - 奇数个节点（3、5、7）
   - 通过 Raft 选主，只有一个 Leader 处理事务
   - 网络分区时，少数节点无法选主，停止服务

2. 使用 db 模式存储：
   - 所有 TC 节点共享同一数据库
   - 通过数据库行锁防止并发冲突
   - 即使脑裂，数据库锁保证一致性

3. 监控告警：
   - 监控 TC 节点数量变化
   - 网络分区时告警，人工介入
```

## 13.3 undolog 表过大处理

### 问题描述

```
表现：
  undo_log 表持续增大，磁盘空间不足
  查询 undo_log 表变慢，影响 AT 模式性能

原因：
  1. 全局事务提交后，undo_log 的异步删除任务积压
  2. Seata 客户端异常退出，删除任务未执行
  3. 未配置定期清理任务
```

### 解决方案

```sql
-- 手动清理已完成的 undo_log（log_status=1 表示已处理）
DELETE FROM undo_log
WHERE log_status = 1
  AND log_created < DATE_SUB(NOW(), INTERVAL 7 DAY)
LIMIT 1000;  -- 分批删除，避免锁表

-- 查看 undo_log 状态分布
SELECT log_status, COUNT(*) as cnt, MIN(log_created), MAX(log_created)
FROM undo_log
GROUP BY log_status;
```

```java
    public void cleanUndoLog() {
        int deleted = 0;
        int totalDeleted = 0;
        do {
            deleted = jdbcTemplate.update(
                "DELETE FROM undo_log " +
                "WHERE log_status = 1 " +
                "  AND log_created < DATE_SUB(NOW(), INTERVAL 7 DAY) " +
                "LIMIT 1000"
            );
            totalDeleted += deleted;
            log.info("清理 undo_log，本批次删除 {} 条", deleted);
            if (deleted > 0) {
                try {
                    Thread.sleep(100);  // 每批次间隔100ms，减轻DB压力
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        } while (deleted >= 1000);
        log.info("undo_log 清理完成，共删除 {} 条", totalDeleted);
    }
}
```

# Part 14: 常见面试题 FAQ

## Q1：什么是分布式事务？为什么微服务需要分布式事务？

**答：**

分布式事务是指涉及多个独立数据库或服务节点的事务，需要保证所有参与者的操作要么全部成功，要么全部失败。

微服务架构中，每个服务有独立的数据库（Database Per Service 原则），一个业务操作（如电商下单）需要同时操作订单库、库存库、账户库。Spring 的 `@Transactional` 只能管理单一数据库连接内的事务，无法跨多个数据库，因此需要分布式事务解决方案。

---

## Q2：CAP 理论是什么？分布式系统中为什么只能选 CP 或 AP？

**答：**

CAP 理论：在分布式系统中，一致性（C）、可用性（A）、分区容错性（P）三者不能同时满足。

原因：网络分区（P）在真实的分布式系统中是不可避免的（网络延迟、节点故障等都会导致分区）。因此 P 必须被满足，实际上只有两种选择：

- **CP**（ZooKeeper）：发生分区时，拒绝写入（停服）来保证数据一致性
- **AP**（Eureka、Cassandra）：发生分区时，仍然提供服务，但可能返回旧数据

---

## Q3：2PC 有哪些问题？3PC 解决了哪些问题？

**答：**

**2PC 的三大问题：**
1. **同步阻塞**：所有参与者在等待协调者期间持有资源锁
2. **单点故障**：协调者崩溃后，参与者无法继续，陷入阻塞
3. **数据不一致**：协调者发出 COMMIT 后崩溃，部分参与者提交，部分未提交

**3PC 改进：**
1. 增加 CanCommit 阶段（不锁资源），减少不必要的锁定时间
2. 增加超时机制，参与者超时后可自行决策（减少无限阻塞）

**3PC 仍存在的问题：**
- 仍然无法彻底解决数据不一致（网络分区 + 超时自行提交可能导致不一致）

---

## Q4：Seata 的三大角色（TC、TM、RM）各自的职责是什么？

**答：**

- **TC（Transaction Coordinator，事务协调者）**：Seata Server，维护全局事务和分支事务的状态，驱动全局提交/回滚，管理全局锁
- **TM（Transaction Manager，事务管理器）**：`@GlobalTransactional` 标注的服务，负责开启全局事务、提交或回滚全局事务
- **RM（Resource Manager，资源管理器）**：每个参与全局事务的服务（DataSourceProxy），向 TC 注册分支事务，执行提交/回滚

---

## Q5：Seata AT 模式的工作原理是什么？

**答：**

AT 模式基于两阶段提交，但做了关键优化：

**第一阶段：**
1. 拦截业务 SQL，在执行前查询 before image（加 FOR UPDATE）
2. 执行业务 SQL
3. 查询 after image
4. 生成 undo_log = {before image, after image}
5. 向 TC 注册分支事务，申请全局锁（防止并发修改）
6. **本地事务立即提交**（业务数据 + undo_log 原子提交）
7. 向 TC 汇报分支状态

**第二阶段-提交：**
- 异步删除 undo_log（不在主链路上，性能极高）

**第二阶段-回滚：**
- 根据 undo_log 的 before image 生成反向 SQL（如 UPDATE balance=1000）
- 执行反向 SQL 恢复数据

核心创新：**Phase 1 就提交本地事务，大幅减少锁持有时间**

---

## Q6：AT 模式和 2PC 有什么区别？

**答：**

| 对比维度 | 传统 2PC | Seata AT |
|----------|---------|----------|
| 第一阶段 | Prepare（持锁不提交） | 业务SQL执行，立即提交本地事务 |
| 锁持有时间 | 很长（Phase1到Phase2全程） | 极短（本地事务提交即释放） |
| 第二阶段 | 发送 Commit/Rollback | 提交：异步删除undo_log；回滚：执行反向SQL |
| 隔离机制 | 数据库行锁 | 全局锁（存储在 TC） |
| 性能 | 差（锁持有时间长） | 好（锁持有时间短） |

---

## Q7：Seata AT 模式如何防止脏写？全局锁的作用是什么？

**答：**

在 AT 模式中，第一阶段提交后本地行锁已释放，但全局锁仍然由 TC 持有。

**全局锁的作用：**
- 防止其他 Seata 事务在当前全局事务未结束前修改相同数据
- 当两个 Seata 事务竞争同一行数据时，通过全局锁序列化访问

**限制：**
- 全局锁只对 Seata 管理的事务有效
- 如果有代码绕过 DataSourceProxy 直接操作数据库，全局锁无法拦截

---

## Q8：什么是 TCC 模式？Try/Confirm/Cancel 各自做什么？

**答：**

TCC（Try-Confirm-Cancel）是一种应用层分布式事务模式：

- **Try**：预留资源（如冻结100元），做业务检查和资源预留，不执行真正业务
- **Confirm**：所有 Try 成功后，确认执行业务（如扣减冻结金额，完成转账）
- **Cancel**：Try 失败时，释放预留资源（如解冻100元）

TCC 需要业务代码实现三个方法，侵入性较高，但性能好（无 undo_log，资源持锁时间短）。

---

## Q9：TCC 模式如何处理幂等性、空回滚和悬挂问题？

**答：**

这三个问题是 TCC 开发中最容易踩坑的地方：

**幂等性：**
- 问题：TC 可能因网络超时多次调用 Confirm/Cancel
- 解决：在业务中记录 TCC 执行日志表（xid + action），每次执行前检查是否已执行

**空回滚：**
- 问题：Try 因网络问题未执行，TC 先发了 Cancel
- 解决：Cancel 时检查 Try 日志是否存在，不存在则直接返回成功（空回滚）

**悬挂：**
- 问题：Try 因网络堵塞延迟到达，此时全局事务已回滚，Cancel 已执行，Try 再来执行会导致资源永久冻结
- 解决：Try 时检查 Cancel 日志是否已存在，已存在则拒绝 Try 执行（返回 false）

---

## Q10：Seata Saga 模式是什么？适合什么场景？

**答：**

Saga 模式将长事务拆分为一系列本地事务，每个本地事务有对应的补偿事务。

**正向流程**：T1 -> T2 -> T3 -> ... -> Tn（顺序执行）
**补偿流程**：如果 Ti 失败，则逆序执行 C(i-1) -> ... -> C1 恢复数据

**适用场景：**
- 长流程业务（如电商退款：取消订单->退货->退积分->退款）
- 遗留系统集成（无法改造成 TCC 的第三方服务）
- 微服务跨系统协调（无法共享数据库）

**缺点：**
- 中间状态可见（隔离性弱），其他事务可能读到不一致的数据
- 补偿逻辑复杂，某些操作不可逆（如已发货的订单）

---

## Q11：Seata XA 模式和 AT 模式有什么区别？

**答：**

| 维度 | AT 模式 | XA 模式 |
|------|---------|---------|
| 原理 | 两阶段+undo补偿 | 数据库原生XA协议 |
| 一致性 | 最终一致（依赖全局锁） | 强一致（数据库保证ACID） |
| 性能 | 好（Phase1提交）| 差（Phase1持锁） |
| 隔离性 | 需要 FOR UPDATE | 数据库原生隔离 |
| 适用场景 | 绝大多数场景 | 需要强一致性的核心业务 |

---

## Q12：什么情况下 AT 模式的回滚会失败？如何处理？

**答：**

AT 模式回滚失败的原因：**脏数据检查失败**（after image 与 current image 不一致）

场景：有代码绕过 Seata DataSourceProxy 直接修改了数据库，导致当前数据与 undo_log 中记录的 after image 不一致。

处理方式：
1. **报警人工处理**（推荐）：记录异常日志，发送告警，由 DBA 人工恢复数据
2. **自定义 FailureHandler**：实现 `io.seata.tm.api.FailureHandler` 接口，自定义处理逻辑
3. **预防**：确保所有数据库操作都通过 DataSourceProxy，不绕过代理

## Q14：Seata Server 支持哪些存储模式？生产环境推荐用哪种？

- **file 模式**：内存存储，无持久化，适合本地测试
- **db 模式**：MySQL 持久化，支持集群，**生产环境推荐**

## Q15：AT 模式的 undo_log 表有什么作用？什么时候创建？

**答：**

`undo_log` 表存储 AT 模式的回滚日志（before image + after image），是 AT 模式实现自动补偿的核心。

**什么时候创建：**
- 每个参与 AT 模式全局事务的数据库都需要创建此表
- 创建脚本：`script/client/at/db/mysql.sql`

**什么时候写入/删除：**
- 写入：Phase 1，与业务数据在同一个本地事务中原子写入
- 删除：Phase 2 Commit，异步删除（不在主链路上）
- 使用：Phase 2 Rollback，读取 before image 生成反向 SQL

## Q16：Seata 全局事务的 XID 是什么？格式是什么？

**答：**

XID（eXternal Transaction ID）是全局事务的唯一标识符。

**格式**：`{TC IP}:{TC Port}:{事务序列号}`

**示例**：`192.168.1.100:8091:2023050812345678`

**XID 的传播：**
- TM 开启全局事务时，TC 生成 XID
- TM 将 XID 绑定到当前线程（`RootContext.bind(xid)`）
- Feign 调用时，通过 HTTP Header `TX_XID` 传递到下游
- 下游 RM 服务从 Header 读取 XID，绑定到本地线程
- DataSourceProxy 拦截 SQL 时，读取线程中的 XID，注册为对应全局事务的分支

## Q17：如何配置 Seata Server 集群高可用？

## Q19：在分布式事务中，如何处理超时问题？

**答：**

**超时场景分析：**

```
场景1：全局事务超时（TM端）
  配置：@GlobalTransactional(timeoutMills = 30000)
  处理：超时后 TC 自动将事务标记为 TimeoutRollbacking，驱动所有 RM 回滚

场景2：RM 执行超时（如账户服务响应慢）
  问题：Feign 调用超时抛出异常，TM 感知到异常后触发全局回滚
  处理：全局事务回滚，已执行的 RM 分支回滚

场景3：TC 与 RM 之间的 Phase 2 通知超时
  处理：TC 会重试，直到 RM 确认（最大重试次数可配置）

最佳实践：
  1. 设置合理的全局事务超时时间（不要太短也不要太长）
  2. Feign 超时时间应小于全局事务超时时间
  3. 核心服务添加快速失败机制，避免长时间阻塞
```

## Q20：Seata 和 RocketMQ 事务消息有什么区别？如何选择？

**答：**

| 维度 | Seata（AT/TCC/XA） | RocketMQ 事务消息 |
|------|-----------------|-----------------|
| 一致性 | 强一致/最终一致 | 最终一致 |
| 同步/异步 | 同步 | 异步 |
| 适用场景 | 需要同步感知结果的场景 | 允许异步处理的场景 |
| 回滚方式 | 自动回滚（undo/反向SQL） | 消息不发送（Rollback Message） |
| 性能 | 相对较低（同步等待） | 高（异步处理） |
| 复杂度 | 中等（框架处理）| 中等（需要事务监听器） |

**选择建议：**
- 需要同步返回结果、强一致性 → 选 **Seata**
- 可以异步处理、追求高性能、通知型场景 → 选 **RocketMQ 事务消息**
- 例如：下单扣库存扣余额（需要同步结果）→ Seata；下单后发送邮件通知 → RocketMQ

## Q21：Seata 性能瓶颈在哪里？如何优化？

**答：**

**主要性能瓶颈：**

1. **TC 是中心化单点**：所有 RM 都要与 TC 通信，TC 吞吐量上限决定整体上限
2. **全局锁竞争**：高并发下热点数据的全局锁竞争激烈
3. **undo_log 写入**：每个 DML 操作都需要额外的 undo_log 写入
4. **网络 RPC 开销**：每个分支事务需要多次与 TC 的网络通信

**优化方案：**

```
1. TC 集群化：多个 TC 节点，通过 Nacos 负载均衡分散压力

2. 缩短全局事务范围：
   减少事务内的操作数量，减少锁持有时间和 RPC 次数

3. 异步化：
   将非核心操作（如日志记录、消息推送）移出全局事务

4. 使用 TCC 替代 AT：
   TCC 无需 undo_log，锁持有时间更短（Try 阶段用行锁，Confirm 阶段无锁）

5. 批量操作优化：
   使用批量 INSERT/UPDATE，减少 undo_log 条数

6. TC 存储优化：
   使用 Redis 存储模式（比 db 模式性能高 3-5 倍）
```

## Q22：如何实现分布式事务的最终一致性监控和报警？

**答：**

```java
// 监控全局事务成功率和耗时
@Aspect
@Component
@Slf4j
public class GlobalTransactionMonitor {

    @Autowired
    private MeterRegistry meterRegistry;

    @Around("@annotation(globalTransactional)")
    public Object monitor(ProceedingJoinPoint pjp,
                          GlobalTransactional globalTransactional) throws Throwable {
        long startTime = System.currentTimeMillis();
        String txName = globalTransactional.name();

        try {
            Object result = pjp.proceed();
            // 记录成功
            meterRegistry.counter("seata.transaction.success",
                "txName", txName).increment();
            return result;
        } catch (Exception e) {
            // 记录失败
            meterRegistry.counter("seata.transaction.failure",
                "txName", txName,
                "exception", e.getClass().getSimpleName()
            ).increment();
            // 发送告警
            sendAlert(txName, e);
            throw e;
        } finally {
            // 记录耗时
            meterRegistry.timer("seata.transaction.duration",
                "txName", txName
            ).record(System.currentTimeMillis() - startTime,
                     TimeUnit.MILLISECONDS);
        }
    }

    private void sendAlert(String txName, Exception e) {
        // 发送钉钉/企业微信/短信告警
        log.error("全局事务失败告警！txName={}, error={}", txName, e.getMessage());
    }
}

// 监控 undo_log 积压
@Scheduled(fixedDelay = 60000)
public void checkUndoLogBacklog() {
    long count = undoLogMapper.countPending();  // log_status=0 的数量
    if (count > 10000) {
        log.warn("undo_log 积压告警：积压 {} 条，可能存在分支事务未完成", count);
        sendUndoLogAlert(count);
    }
}
```
