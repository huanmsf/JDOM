### 1.2 容量估算

容量估算是系统设计的基础，以下是常用数据：

#### 时间单位换算
```
1 day    = 86,400 秒  ≈ 10^5 秒
1 month  = 2,592,000 秒 ≈ 2.5 * 10^6 秒
1 year   = 31,536,000 秒 ≈ 3 * 10^7 秒
```

#### 存储单位换算
```
1 KB = 1,024 Bytes    ≈ 10^3 Bytes
1 MB = 1,048,576 Bytes ≈ 10^6 Bytes
1 GB = 10^9 Bytes
1 TB = 10^12 Bytes
1 PB = 10^15 Bytes
```

#### 典型QPS估算示例（短链系统）
```
假设：
  - DAU（日活用户）= 1亿
  - 每个用户每天平均创建0.1个短链，点击2次
  - 写QPS = 1亿 * 0.1 / 86400 ≈ 116 QPS
  - 读QPS = 1亿 * 2 / 86400  ≈ 2315 QPS
  - 读写比 ≈ 20:1

存储估算（5年）：
  - 每天新增短链 = 1亿 * 0.1 = 1000万条
  - 5年总量 = 1000万 * 365 * 5 ≈ 180亿条
  - 每条记录约500字节
  - 总存储 = 180亿 * 500B ≈ 9TB
```

### 1.4 可扩展性设计

#### 水平扩展 vs 垂直扩展
```
垂直扩展 (Scale Up)          水平扩展 (Scale Out)
+------------------+         +----+ +----+ +----+
|                  |         | S1 | | S2 | | S3 |
|   大型服务器      |   vs    +----+ +----+ +----+
|  (更多CPU/内存)   |         负载均衡 + 无状态服务
+------------------+
  上限有限，成本高           理论无上限，成本线性
```

#### 数据库扩展策略
```
1. 读写分离
   写 -> [Master]
   读 -> [Slave1] [Slave2] [Slave3]

2. 分库分表
   用户ID % 4 = 0 -> DB0
   用户ID % 4 = 1 -> DB1
   用户ID % 4 = 2 -> DB2
   用户ID % 4 = 3 -> DB3

3. NewSQL (TiDB / CockroachDB)
   - 自动水平分片
   - 兼容MySQL协议
   - 强一致性 + 高可用
```

### 2.3 Base62 编码

```
字符集: a-z A-Z 0-9 (共62个字符)
6位Base62可表示: 62^6 = 568亿 种组合

ID -> Base62:
  1234567890 -> "1LY7VK"

算法：
  while (id > 0):
    result = CHARS[id % 62] + result
    id = id / 62
```

#### Java 实现

```java
@Component
public class Base62Encoder {
    private static final String CHARS =
        "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";

    public String encode(long id) {
        if (id == 0) return String.valueOf(CHARS.charAt(0));
        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(CHARS.charAt((int)(id % 62)));
            id /= 62;
        }
        return sb.reverse().toString();
    }

    public long decode(String shortCode) {
        long result = 0;
        for (char c : shortCode.toCharArray()) {
            result = result * 62 + CHARS.indexOf(c);
        }
        return result;
    }
}
```

### 2.4 ID 生成：号段模式

为避免每次生成ID都访问数据库，使用号段模式批量获取ID段：

```
数据库表：
+----------+-----------+-----------+----------+
| biz_tag  | max_id    | step      | version  |
+----------+-----------+-----------+----------+
| url      | 1000000   | 1000      | 5        |
+----------+-----------+-----------+----------+

每次取号段：
UPDATE id_alloc SET max_id = max_id + step, version = version + 1
WHERE biz_tag = 'url' AND version = #{version}

返回 [max_id - step, max_id) 区间的ID供本地使用
```

#### Java 实现

```java
@Service
public class SegmentIdGenerator {
    @Autowired
    private IdAllocMapper idAllocMapper;

    private volatile long currentId;
    private volatile long maxId;
    private final Object lock = new Object();

    public long nextId(String bizTag) {
        synchronized (lock) {
            if (currentId >= maxId) {
                loadNextSegment(bizTag);
            }
            return currentId++;
        }
    }

    private void loadNextSegment(String bizTag) {
        // 乐观锁重试
        for (int i = 0; i < 3; i++) {
            IdAllocDO record = idAllocMapper.selectByBizTag(bizTag);
            long newMaxId = record.getMaxId() + record.getStep();
            int rows = idAllocMapper.updateMaxId(bizTag, newMaxId, record.getVersion());
            if (rows == 1) {
                currentId = record.getMaxId();
                maxId = newMaxId;
                return;
            }
        }
        throw new RuntimeException("获取ID号段失败，重试耗尽");
    }
}
```

### 2.5 短链创建 API

```java
@RestController
@RequestMapping("/api/url")
public class UrlController {

    @Autowired
    private UrlService urlService;

    @PostMapping("/shorten")
    public ResponseEntity<ShortenResponse> shorten(@RequestBody ShortenRequest req) {
        // 1. 参数校验
        if (!isValidUrl(req.getLongUrl())) {
            return ResponseEntity.badRequest().build();
        }
        // 2. 检查是否已存在（幂等）
        String existing = urlService.findByLongUrl(req.getLongUrl());
        if (existing != null) {
            return ResponseEntity.ok(new ShortenResponse(existing));
        }
        // 3. 生成短链
        String shortCode = urlService.createShortUrl(req.getLongUrl(), req.getExpireAt());
        String shortUrl = "https://t.co/" + shortCode;
        return ResponseEntity.ok(new ShortenResponse(shortUrl));
    }
}

@Service
public class UrlService {
    @Autowired
    private Base62Encoder encoder;
    @Autowired
    private SegmentIdGenerator idGenerator;
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    @Autowired
    private UrlMapper urlMapper;

    public String createShortUrl(String longUrl, LocalDateTime expireAt) {
        long id = idGenerator.nextId("url");
        String shortCode = encoder.encode(id);

        // 持久化到MySQL
        UrlDO urlDO = new UrlDO();
        urlDO.setShortCode(shortCode);
        urlDO.setLongUrl(longUrl);
        urlDO.setExpireAt(expireAt);
        urlDO.setCreateTime(LocalDateTime.now());
        urlMapper.insert(urlDO);

        // 写入Redis缓存（设置过期时间）
        String cacheKey = "url:" + shortCode;
        if (expireAt != null) {
            long ttl = ChronoUnit.SECONDS.between(LocalDateTime.now(), expireAt);
            redisTemplate.opsForValue().set(cacheKey, longUrl, ttl, TimeUnit.SECONDS);
        } else {
            redisTemplate.opsForValue().set(cacheKey, longUrl, 7, TimeUnit.DAYS);
        }
        return shortCode;
    }

    public String getLongUrl(String shortCode) {
        // 1. 先查Redis
        String cacheKey = "url:" + shortCode;
        String longUrl = redisTemplate.opsForValue().get(cacheKey);
        if (longUrl != null) return longUrl;

        // 2. 再查MySQL
        UrlDO urlDO = urlMapper.selectByShortCode(shortCode);
        if (urlDO == null) return null;

        // 3. 回填缓存
        redisTemplate.opsForValue().set(cacheKey, urlDO.getLongUrl(), 7, TimeUnit.DAYS);
        return urlDO.getLongUrl();
    }
}
```

### 2.6 短链跳转：301 vs 302

```
301 永久重定向：
  - 浏览器会缓存，后续直接跳转不经过服务器
  - 优点：减少服务器压力
  - 缺点：无法统计点击次数，无法修改目标URL

302 临时重定向：
  - 每次都经过服务器
  - 优点：可统计、可修改
  - 缺点：服务器压力较大

推荐：302（需要统计时），301（纯跳转场景）
```

```java
@GetMapping("/{shortCode}")
public ResponseEntity<Void> redirect(@PathVariable String shortCode,
                                      HttpServletRequest request) {
    String longUrl = urlService.getLongUrl(shortCode);
    if (longUrl == null) {
        return ResponseEntity.notFound().build();
    }
    // 异步记录访问日志
    asyncLogService.logVisit(shortCode, request.getRemoteAddr());
    return ResponseEntity.status(HttpStatus.FOUND)  // 302
        .header("Location", longUrl)
        .build();
}
```

### 2.7 防刷限流

使用 Redis + Lua 脚本实现滑动窗口限流：

```java
@Component
public class RateLimiter {
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String RATE_LIMIT_SCRIPT =
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +
        "local window = tonumber(ARGV[2]) " +
        "local now = tonumber(ARGV[3]) " +
        "redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000) " +
        "local count = redis.call('ZCARD', key) " +
        "if count < limit then " +
        "  redis.call('ZADD', key, now, now) " +
        "  redis.call('EXPIRE', key, window) " +
        "  return 1 " +
        "else " +
        "  return 0 " +
        "end";

    public boolean isAllowed(String ip, int limit, int windowSeconds) {
        String key = "rate_limit:create:" + ip;
        long now = System.currentTimeMillis();
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(RATE_LIMIT_SCRIPT, Long.class),
            Collections.singletonList(key),
            String.valueOf(limit),
            String.valueOf(windowSeconds),
            String.valueOf(now)
        );
        return Long.valueOf(1L).equals(result);
    }
}
```

### 3.3 Redis Lua 库存扣减

Lua 脚本保证原子性，防止超卖：

```lua
-- deduct_stock.lua
-- KEYS[1]: 库存key，如 "stock:activity:1001"
-- ARGV[1]: 扣减数量
-- 返回: 1成功，-1库存不足，-2活动未开始/已结束

local stock_key = KEYS[1]
local deduct = tonumber(ARGV[1])

-- 检查库存是否存在
local stock = redis.call('GET', stock_key)
if not stock then
    return -2
end

stock = tonumber(stock)
if stock < deduct then
    return -1
end

-- 原子扣减
redis.call('DECRBY', stock_key, deduct)
return 1
```

```java
@Service
public class SeckillService {

    private static final String DEDUCT_STOCK_SCRIPT =
        "local stock = redis.call('GET', KEYS[1])
" +
        "if not stock then return -2 end
" +
        "stock = tonumber(stock)
" +
        "if stock < tonumber(ARGV[1]) then return -1 end
" +
        "redis.call('DECRBY', KEYS[1], ARGV[1])
" +
        "return 1";

    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    @Autowired
    private KafkaTemplate<String, SeckillOrderEvent> kafkaTemplate;

    public SeckillResult doSeckill(Long userId, Long activityId, int quantity) {
        // 1. 用户级别限流（每人限购1件）
        String userLimitKey = "seckill:user:" + userId + ":activity:" + activityId;
        Boolean alreadyBought = redisTemplate.opsForValue()
            .setIfAbsent(userLimitKey, "1", 24, TimeUnit.HOURS);
        if (!Boolean.TRUE.equals(alreadyBought)) {
            return SeckillResult.fail("每人限购一件");
        }

        // 2. Redis Lua 原子扣减库存
        String stockKey = "stock:activity:" + activityId;
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(DEDUCT_STOCK_SCRIPT, Long.class),
            Collections.singletonList(stockKey),
            String.valueOf(quantity)
        );

        if (result == null || result == -2L) {
            // 回滚用户限购标记
            redisTemplate.delete(userLimitKey);
            return SeckillResult.fail("活动不存在");
        }
        if (result == -1L) {
            redisTemplate.delete(userLimitKey);
            return SeckillResult.fail("库存不足");
        }

        // 3. 发送MQ消息，异步创建订单
        String orderId = generateOrderId(userId, activityId);
        SeckillOrderEvent event = new SeckillOrderEvent(orderId, userId, activityId, quantity);
        kafkaTemplate.send("seckill-orders", String.valueOf(userId), event);

        return SeckillResult.success(orderId);
    }
}
```

### 3.4 MQ 异步下单

```java
@Component
@KafkaListener(topics = "seckill-orders", groupId = "seckill-order-group")
public class SeckillOrderConsumer {

    @Autowired
    private OrderService orderService;
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @KafkaHandler
    public void handleSeckillOrder(SeckillOrderEvent event) {
        // 幂等检查（防止重复消费）
        String idempotentKey = "seckill:order:created:" + event.getOrderId();
        Boolean isNew = redisTemplate.opsForValue()
            .setIfAbsent(idempotentKey, "1", 24, TimeUnit.HOURS);
        if (!Boolean.TRUE.equals(isNew)) {
            log.warn("重复消息，跳过: {}", event.getOrderId());
            return;
        }

        try {
            // 创建数据库订单
            orderService.createSeckillOrder(event);
            log.info("秒杀订单创建成功: {}", event.getOrderId());
        } catch (Exception e) {
            // 失败回滚：归还库存
            String stockKey = "stock:activity:" + event.getActivityId();
            redisTemplate.opsForValue().increment(stockKey, event.getQuantity());
            redisTemplate.delete(idempotentKey);
            throw e;  // 重新入队重试
        }
    }
}
```

### 3.5 超卖防护分析

```
超卖场景分析：

方案一：MySQL行锁（不推荐高并发）
  UPDATE stock SET count = count - 1
  WHERE activity_id = ? AND count > 0
  问题：高并发下行锁竞争严重，TPS极低

方案二：Redis Lua（推荐）
  优点：
    - Lua脚本原子执行，无竞争条件
    - Redis单线程，天然串行
    - 高性能：10万+ TPS
  注意：
    - 需要Redis主从同步（主挂时从节点数据可能丢失）
    - 建议使用RedLock或主从切换后补偿机制

方案三：消息队列串行化（超大规模）
  - 所有扣减请求入队
  - 单个消费者串行处理
  - 优点：绝对不超卖
  - 缺点：延迟高，队列积压
```

### 3.6 活动预热

```java
@Service
public class SeckillInitService {

    @Autowired
    private ActivityMapper activityMapper;
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // 活动开始前5分钟预热
    @Scheduled(fixedDelay = 60000)
    public void preloadActivities() {
        List<ActivityDO> activities = activityMapper.selectUpcomingActivities(5);
        for (ActivityDO activity : activities) {
            String stockKey = "stock:activity:" + activity.getId();
            // 预加载库存到Redis
            redisTemplate.opsForValue().set(stockKey,
                String.valueOf(activity.getStock()),
                activity.getDurationMinutes() + 30,
                TimeUnit.MINUTES);

            // 预加载活动信息（减少DB查询）
            String infoKey = "activity:info:" + activity.getId();
            redisTemplate.opsForValue().set(infoKey, activity, 1, TimeUnit.HOURS);

            log.info("活动预热完成: activityId={}, stock={}", activity.getId(), activity.getStock());
        }
    }
}
```

```java
public class WebSocketConnectionManager {

        new ConcurrentHashMap<>();

    private RedisTemplate<String, String> redisTemplate;

        String key = "ws:user:" + userId;
        redisTemplate.opsForValue().set(key, nodeId, 30, TimeUnit.MINUTES);
        log.info("用户{}连接到节点{}", userId, nodeId);
    }

    public void onDisconnect(Long userId) {
        localConnections.remove(userId);
        redisTemplate.delete("ws:user:" + userId);
        log.info("用户{}断开连接", userId);
    }

    public boolean sendToLocalUser(Long userId, String message) {
        WebSocketSession session = localConnections.get(userId);
        if (session != null && session.isOpen()) {
            try {
                session.sendMessage(new TextMessage(message));
                return true;
            } catch (IOException e) {
                log.error("发送消息失败", e);
                onDisconnect(userId);
            }
        }
        return false;
    }

    public String getUserNodeId(Long userId) {
        return redisTemplate.opsForValue().get("ws:user:" + userId);
    }
```

```java
public class MessageRouteService {

    private WebSocketConnectionManager connectionManager;
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    @Autowired
    private KafkaTemplate<String, PushMessage> kafkaTemplate;

    public void routeMessage(PushMessage message) {
        Long targetUserId = message.getTargetUserId();

        // 1. 查询目标用户在哪个节点
        String targetNodeId = connectionManager.getUserNodeId(targetUserId);

        if (targetNodeId == null) {
            // 用户离线，存入离线消息队列
            saveOfflineMessage(message);
            return;
        }

        String currentNodeId = System.getenv("NODE_ID");
        if (targetNodeId.equals(currentNodeId)) {
            // 2. 目标在本节点，直接推送
            boolean sent = connectionManager.sendToLocalUser(targetUserId,
                JSON.toJSONString(message));
            if (!sent) {
                saveOfflineMessage(message);
            }
        } else {
            // 3. 目标在其他节点，通过Redis Pub/Sub转发
            String channel = "push:node:" + targetNodeId;
            redisTemplate.convertAndSend(channel, JSON.toJSONString(message));
        }
    }

    private void saveOfflineMessage(PushMessage message) {
        // 存入Kafka，消费后持久化到MySQL
        kafkaTemplate.send("offline-messages", message);
    }
}
```

```java
    private FollowMapper followMapper;

    private KafkaTemplate<String, FeedEvent> kafkaTemplate;

    public void fanoutPost(Long authorId, Long postId, long timestamp) {
        // 异步推送，避免阻塞
        kafkaTemplate.send("feed-fanout", new FeedEvent(authorId, postId, timestamp));
    }

    // Feed箱：使用Redis ZSet，score=时间戳，member=postId
    // key: "feed:{userId}"
    public void pushToFollower(Long followerId, Long postId, long timestamp) {
        String feedKey = "feed:" + followerId;
        redisTemplate.opsForZSet().add(feedKey, postId.toString(), timestamp);
        // 只保留最近500条（内存限制）
        redisTemplate.opsForZSet().removeRange(feedKey, 0, -501);
        // 设置过期时间
        redisTemplate.expire(feedKey, 7, TimeUnit.DAYS);
    }

    // 读取Feed流（游标分页）
    public FeedPage getFeed(Long userId, Long cursor, int pageSize) {
        String feedKey = "feed:" + userId;
        double minScore = cursor == null ? Double.MAX_VALUE : (double) cursor;

        // 按score（时间戳）倒序，从cursor开始取pageSize+1条
        Set<ZSetOperations.TypedTuple<Object>> items =
            redisTemplate.opsForZSet().reverseRangeByScoreWithScores(
                feedKey, 0, minScore - 1, 0, pageSize + 1
            );

        List<Long> postIds = new ArrayList<>();
        Long nextCursor = null;

        if (items != null) {
            List<ZSetOperations.TypedTuple<Object>> list = new ArrayList<>(items);
            for (int i = 0; i < Math.min(list.size(), pageSize); i++) {
                postIds.add(Long.valueOf(list.get(i).getValue().toString()));
            }
            if (list.size() > pageSize) {
                nextCursor = list.get(pageSize - 1).getScore().longValue();
            }
        }

        return new FeedPage(postIds, nextCursor, nextCursor == null);
    }
}
```

### 6.4 威尔逊置信区间热评排序

```
威尔逊得分公式（考虑样本量的置信区间下界）：
  score = (p + z²/2n - z*sqrt(p(1-p)/n + z²/4n²)) / (1 + z²/n)

其中：
  p = 点赞数 / (点赞数 + 踩数)  （正向比例）
  n = 总投票数
  z = 1.96（95%置信度对应的z值）

优势：
  - 10赞0踩 vs 100赞10踩，后者更可信
  - 避免纯按点赞数排序（新评论没机会）
  - Reddit、Hacker News采用类似算法
```

```java
@Component
public class WilsonScoreCalculator {

    private static final double Z = 1.96;  // 95% 置信度

    public double calculate(long upvotes, long downvotes) {
        long n = upvotes + downvotes;
        if (n == 0) return 0.0;

        double p = (double) upvotes / n;
        double z2 = Z * Z;

        double numerator = p + z2 / (2 * n)
            - Z * Math.sqrt(p * (1 - p) / n + z2 / (4 * n * n));
        double denominator = 1 + z2 / n;

        return numerator / denominator;
    }
}

@Service
public class HotCommentRankService {

    @Autowired
    private WilsonScoreCalculator calculator;
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // 评论点赞后更新热度分
    public void updateHotScore(Long postId, Long commentId, long upvotes, long downvotes) {
        double score = calculator.calculate(upvotes, downvotes);
        String key = "hot_comments:" + postId;
        redisTemplate.opsForZSet().add(key, commentId.toString(), score);
    }

    // 查询热评（ZSet按score倒序）
    public List<Long> getHotComments(Long postId, int topN) {
        String key = "hot_comments:" + postId;
        Set<Object> result = redisTemplate.opsForZSet()
            .reverseRange(key, 0, topN - 1);
        return result.stream()
            .map(o -> Long.valueOf(o.toString()))
            .collect(Collectors.toList());
    }
}
```

## Part 7: 分布式ID生成器 {#part-7}

### 7.1 雪花算法（Snowflake）

```
64位结构：
+----------+-------------------+-------------------+-------------------+
| 符号位(1) | 时间戳(41位)       | 机器ID(10位)       | 序列号(12位)       |
+----------+-------------------+-------------------+-------------------+
  固定为0   毫秒级时间戳-纪元    最多1024台机器       每毫秒4096个ID

最大QPS = 1024 * 4096 * 1000 = 4.2 * 10^9 /s

时间范围：41位时间戳，约69年
```

### 7.2 完整 Java 实现（含时钟回拨处理）

```java
@Component
public class SnowflakeIdGenerator {

    // 纪元（2020-01-01 00:00:00 UTC）
    private static final long EPOCH = 1577836800000L;

    // 各字段位数
    private static final long WORKER_ID_BITS = 5L;
    private static final long DATACENTER_ID_BITS = 5L;
    private static final long SEQUENCE_BITS = 12L;

    // 最大值
    private static final long MAX_WORKER_ID = ~(-1L << WORKER_ID_BITS);       // 31
    private static final long MAX_DATACENTER_ID = ~(-1L << DATACENTER_ID_BITS); // 31

    // 左移位数
    private static final long WORKER_ID_SHIFT = SEQUENCE_BITS;                // 12
    private static final long DATACENTER_ID_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS; // 17
    private static final long TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS + DATACENTER_ID_BITS; // 22

    private static final long SEQUENCE_MASK = ~(-1L << SEQUENCE_BITS);        // 4095

    private final long workerId;
    private final long datacenterId;

    private long sequence = 0L;
    private long lastTimestamp = -1L;

    // 时钟回拨容忍阈值（5ms内等待，超过则报错）
    private static final long MAX_BACKWARD_MS = 5L;

    public SnowflakeIdGenerator(long workerId, long datacenterId) {
        if (workerId > MAX_WORKER_ID || workerId < 0) {
            throw new IllegalArgumentException("workerId范围：0-" + MAX_WORKER_ID);
        }
        if (datacenterId > MAX_DATACENTER_ID || datacenterId < 0) {
            throw new IllegalArgumentException("datacenterId范围：0-" + MAX_DATACENTER_ID);
        }
        this.workerId = workerId;
        this.datacenterId = datacenterId;
    }

    public synchronized long nextId() {
        long timestamp = currentTime();

        // 时钟回拨检测
        if (timestamp < lastTimestamp) {
            long offset = lastTimestamp - timestamp;
            if (offset <= MAX_BACKWARD_MS) {
                // 容忍范围内，等待时钟追上
                try {
                    Thread.sleep(offset << 1);  // 等待 offset*2 ms
                    timestamp = currentTime();
                    if (timestamp < lastTimestamp) {
                        throw new RuntimeException("时钟回拨超出容忍范围: " + offset + "ms");
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException("时钟等待被中断");
                }
            } else {
                throw new RuntimeException("严重时钟回拨: " + offset + "ms");
            }
        }

        if (lastTimestamp == timestamp) {
            // 同一毫秒内，序列号递增
            sequence = (sequence + 1) & SEQUENCE_MASK;
            if (sequence == 0) {
                // 序列号溢出，等待下一毫秒
                timestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            // 新的毫秒，序列号归零（随机化减少竞争）
            sequence = 0L;
        }

        lastTimestamp = timestamp;

        return ((timestamp - EPOCH) << TIMESTAMP_SHIFT)
            | (datacenterId << DATACENTER_ID_SHIFT)
            | (workerId << WORKER_ID_SHIFT)
            | sequence;
    }

    private long waitNextMillis(long lastTimestamp) {
        long timestamp = currentTime();
        while (timestamp <= lastTimestamp) {
            timestamp = currentTime();
        }
        return timestamp;
    }
```

### 7.3 号段模式双 Buffer

```
双Buffer原理：
  - Buffer A: [1001, 2000]  <- 当前使用
  - Buffer B: [2001, 3000]  <- 预加载完毕，等待切换

当Buffer A使用到50%时，异步加载Buffer B
当Buffer A耗尽时，立即切换到Buffer B
切换后异步加载新的Buffer A

优点：
  - 无需每次生成ID都访问数据库
  - DB故障时，双Buffer可支撑一段时间
  - 切换时无等待，无性能抖动
```

```java
@Component
public class DoubleBufferIdGenerator {

    @Autowired
    private IdAllocMapper idAllocMapper;

    private volatile Segment current;
    private volatile Segment next;
    private volatile boolean isLoadingNext = false;
    private final ReentrantLock lock = new ReentrantLock();

    @PostConstruct
    public void init() {
        current = loadNewSegment();
    }

    public long nextId(String bizTag) {
        // 尝试从当前Buffer取
        long id = current.nextId();
        if (id != -1) {
            // 当使用超过50%时，异步预加载next
            if (current.getUsedPercent() > 50 && next == null && !isLoadingNext) {
                asyncLoadNext(bizTag);
            }
            return id;
        }

        // 当前Buffer耗尽，切换到next
        lock.lock();
        try {
            // 双重检查
            id = current.nextId();
            if (id != -1) return id;

            if (next == null) {
                // next还未加载完，同步加载
                next = loadNewSegment();
            }
            current = next;
            next = null;
            isLoadingNext = false;
            return current.nextId();
        } finally {
            lock.unlock();
        }
    }

    private void asyncLoadNext(String bizTag) {
        isLoadingNext = true;
        CompletableFuture.runAsync(() -> {
            try {
                next = loadNewSegment();
            } catch (Exception e) {
                isLoadingNext = false;
                log.error("异步加载号段失败", e);
            }
        });
    }

    private Segment loadNewSegment() {
        // 乐观锁更新数据库
        for (int i = 0; i < 3; i++) {
            IdAllocDO record = idAllocMapper.selectByBizTag("url");
            long newMaxId = record.getMaxId() + record.getStep();
            int rows = idAllocMapper.updateMaxId("url", newMaxId, record.getVersion());
            if (rows == 1) {
                return new Segment(record.getMaxId(), newMaxId);
            }
        }
        throw new RuntimeException("加载号段失败");
    }
}

class Segment {
    private final AtomicLong current;
    private final long max;

    public Segment(long min, long max) {
        this.current = new AtomicLong(min);
        this.max = max;
    }

    public long nextId() {
        long id = current.getAndIncrement();
        return id < max ? id : -1;
    }

    public int getUsedPercent() {
        return (int) ((current.get() - (max - (max - current.get()))) * 100 / (max - (max - current.get())));
    }
}
```

```java
        "local capacity = tonumber(ARGV[1]) " +      // 桶容量
        "local rate = tonumber(ARGV[2]) " +           // 每秒产生令牌数

        "redis.call('EXPIRE', key, 3600) " +

            String.valueOf(rate),
            String.valueOf(requested),
```

## Part 10: 分布式缓存 {#part-10}

### 10.1 多级缓存架构

```
请求
  |
  v
[L1: 本地缓存 Caffeine]   命中率70%，延迟<1ms
  | miss
  v
[L2: Redis 集群]          命中率25%，延迟1-5ms
  | miss
  v
[L3: 数据库 MySQL]        最后兜底，延迟10-50ms

回填策略：从DB取到数据后，同时写入L1和L2
```

```java
@Service
public class MultiLevelCacheService {

    // L1: 本地缓存 Caffeine
    private final Cache<String, Object> localCache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(1, TimeUnit.MINUTES)   // 本地缓存1分钟，减少一致性问题
        .recordStats()
        .build();

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private ProductMapper productMapper;

    @SuppressWarnings("unchecked")
    public <T> T get(String key, Class<T> type, Supplier<T> dbLoader) {
        // 1. 查L1
        T value = (T) localCache.getIfPresent(key);
        if (value != null) return value;

        // 2. 查L2
        value = (T) redisTemplate.opsForValue().get(key);
        if (value != null) {
            localCache.put(key, value);  // 回填L1
            return value;
        }

        // 3. 查DB（加分布式锁防止缓存击穿）
        String lockKey = "lock:" + key;
        boolean locked = tryLock(lockKey, 5000);
        if (!locked) {
            // 未拿到锁，短暂等待后重试（可能其他线程已回填）
            Thread.sleep(50);
            return get(key, type, dbLoader);
        }
        try {
            // 双重检查
            value = (T) redisTemplate.opsForValue().get(key);
            if (value != null) {
                localCache.put(key, value);
                return value;
            }
            // 真正查DB
            value = dbLoader.get();
            if (value != null) {
                redisTemplate.opsForValue().set(key, value, 30, TimeUnit.MINUTES);
                localCache.put(key, value);
            } else {
                // 缓存空值，防止缓存穿透
                redisTemplate.opsForValue().set(key, "NULL", 5, TimeUnit.MINUTES);
            }
            return value;
        } finally {
            releaseLock(lockKey);
        }
    }
}
```

### 10.2 缓存一致性

```
写操作策略对比：

1. Cache-Aside（旁路缓存）[推荐]
   写：先写DB，再删Cache（不是更新Cache）
   读：先读Cache，miss则读DB，再写Cache
   
   删除vs更新：
   - 删除幂等，并发安全
   - 更新可能导致"并发写覆盖"问题

2. Write-Through（写穿）
   写：同时写DB和Cache
   问题：DB写慢会阻塞

3. Write-Behind（异步写）
   写：先写Cache，异步写DB
   问题：可能丢数据

双删策略（解决读写并发不一致）：
  1. 删除缓存
  2. 写DB
  3. sleep(500ms)    <- 等待可能的旧数据读请求完成
  4. 再次删除缓存   <- 清除可能在步骤2-3间回填的旧缓存
```

```java
@Service
public class CacheConsistencyService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    @Autowired
    private ProductMapper productMapper;

    // Cache-Aside 写操作（延迟双删）
    @Transactional
    public void updateProduct(ProductDO product) {
        String cacheKey = "product:" + product.getId();

        // 1. 先删缓存
        redisTemplate.delete(cacheKey);

        // 2. 写DB
        productMapper.updateById(product);

        // 3. 延迟双删（异步，避免阻塞主线程）
        CompletableFuture.runAsync(() -> {
            try {
                Thread.sleep(500);  // 等待可能的旧读操作完成
                redisTemplate.delete(cacheKey);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
    }
```

#### 缓存击穿（热Key过期瞬间大量请求）

```
问题：某个热Key突然过期，大量请求同时穿透到DB

解决方案：
1. 分布式锁（同一时刻只有一个请求查DB）
2. 逻辑过期（不设TTL，在value中存过期时间）
```

```java
// 逻辑过期方案
@Service
public class LogicalExpireCache {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private final ExecutorService refreshExecutor = Executors.newFixedThreadPool(10);

    public <T> T get(String key, Class<T> type, Supplier<T> dbLoader, long expireSeconds) {
        CacheWrapper<T> wrapper = (CacheWrapper<T>) redisTemplate.opsForValue().get(key);

        if (wrapper == null) {
            // 缓存未命中，同步加载
            T data = dbLoader.get();
            set(key, data, expireSeconds);
            return data;
        }

        if (wrapper.getExpireTime().isAfter(LocalDateTime.now())) {
            // 未过期，直接返回
            return wrapper.getData();
        }

        // 已过期，异步刷新
        String lockKey = "refresh_lock:" + key;
        if (tryLock(lockKey, 10000)) {
            // 拿到锁，异步刷新
            refreshExecutor.submit(() -> {
                try {
                    T freshData = dbLoader.get();
                    set(key, freshData, expireSeconds);
                } finally {
                    releaseLock(lockKey);
                }
            });
        }
        // 返回旧值（可能已过期，但不影响使用）
        return wrapper.getData();
    }

    private <T> void set(String key, T data, long expireSeconds) {
        CacheWrapper<T> wrapper = new CacheWrapper<>();
        wrapper.setData(data);
        wrapper.setExpireTime(LocalDateTime.now().plusSeconds(expireSeconds));
        redisTemplate.opsForValue().set(key, wrapper);  // 不设TTL
    }
}
```

### 10.4 热Key与大Key

```
热Key（Hot Key）：
  单个Key请求量极大，导致单节点CPU飙高
  
  识别：
    - Redis监控：redis-cli --hotkeys
    - 业务层统计（滑动窗口计数）
  
  解决方案：
    1. 本地缓存（L1）：减少Redis请求
    2. Key拆分：product:123 -> product:123:0 ~ product:123:9（随机选一个读）
    3. 读写分离：只读副本分担压力

大Key（Big Key）：
  单个Key存储了大量数据（如String > 10MB，Hash/List/Set > 5000个元素）
  
  危害：
    - 内存不均衡
    - 查询超时（慢查询）
    - 主从同步延迟
  
  解决方案：
    1. 拆分：大Hash按field前缀拆分成多个小Hash
    2. 压缩：序列化时使用Snappy/LZ4压缩
    3. 异步删除：del大Key用unlink命令（非阻塞）
```

```java
@Component
public class HotKeyDetector {

    private final LoadingCache<String, AtomicLong> counters =
        CacheBuilder.newBuilder()
            .expireAfterWrite(1, TimeUnit.MINUTES)
            .build(k -> new AtomicLong(0));

    private static final long HOT_KEY_THRESHOLD = 1000;  // 1分钟内超过1000次即为热Key

    public boolean isHotKey(String key) {
        long count = counters.getUnchecked(key).incrementAndGet();
        return count > HOT_KEY_THRESHOLD;
    }

    // 热Key降级到本地缓存
    private final Cache<String, Object> hotKeyLocalCache =
        Caffeine.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(10, TimeUnit.SECONDS)
            .build();

    public Object getWithHotKeyProtection(String key, Supplier<Object> loader) {
        if (isHotKey(key)) {
            return hotKeyLocalCache.get(key, k -> loader.get());
        }
        return loader.get();
    }
}
```

### 11.3 Redis 分桶方案

```
原理：将库存分散到N个桶，每个请求随机路由到一个桶
     单桶库存 = 总库存 / N
     
     请求 -> 随机选桶[0,N) -> 扣减该桶库存
          -> 桶库存不足 -> 尝试其他桶（兜底）

     N个桶 = N个Redis Key，并发度提升N倍
```

```java
@Service
public class BucketStockService {

    private static final int BUCKET_COUNT = 10;

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final String LUA_SCRIPT =
        "local stock = tonumber(redis.call('GET', KEYS[1]) or '0') " +
        "if stock >= tonumber(ARGV[1]) then " +
        "    redis.call('DECRBY', KEYS[1], ARGV[1]) " +
        "    return 1 " +
        "else " +
        "    return 0 " +
        "end";

    // 初始化分桶库存
    public void initBucketStock(Long skuId, int totalStock) {
        int baseStock = totalStock / BUCKET_COUNT;
        int remainder = totalStock % BUCKET_COUNT;

        for (int i = 0; i < BUCKET_COUNT; i++) {
            String key = "stock:" + skuId + ":bucket:" + i;
            int bucketStock = i == 0 ? baseStock + remainder : baseStock;
            redisTemplate.opsForValue().set(key, String.valueOf(bucketStock));
        }
    }

    // 扣减库存（随机桶 + 兜底重试）
    public boolean deductStock(Long skuId, int quantity) {
        int startBucket = ThreadLocalRandom.current().nextInt(BUCKET_COUNT);

        for (int i = 0; i < BUCKET_COUNT; i++) {
            int bucket = (startBucket + i) % BUCKET_COUNT;
            String key = "stock:" + skuId + ":bucket:" + bucket;
            Long result = redisTemplate.execute(
                new DefaultRedisScript<>(LUA_SCRIPT, Long.class),
                Collections.singletonList(key),
                String.valueOf(quantity)
            );
            if (Long.valueOf(1L).equals(result)) {
                return true;
            }
        }
        return false;  // 所有桶都不足
    }

    // 查询总库存（汇总所有桶）
    public int getTotalStock(Long skuId) {
        int total = 0;
        for (int i = 0; i < BUCKET_COUNT; i++) {
            String key = "stock:" + skuId + ":bucket:" + i;
            String value = redisTemplate.opsForValue().get(key);
            if (value != null) {
                total += Integer.parseInt(value);
            }
        }
        return total;
    }
}
```

### 11.4 Redis预扣 + MQ + 定时对账

```
完整流程：

1. [Redis预扣库存] <- 原子操作，高并发
      |
      | 成功
      v
2. [发送MQ消息] -> 订单ID + SKU + 数量
      |
      v
3. [消费MQ] -> 写DB订单 + 写DB库存流水
      |
      | 失败（异常/超时）
      v
4. [补偿机制] -> 归还Redis库存 + 取消订单

5. [定时对账] (每分钟)
      -> 对比Redis库存 vs DB库存流水
      -> 发现不一致 -> 告警 + 自动修正
```

```java
@Service
@Slf4j
public class StockReconcileService {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    @Autowired
    private StockFlowMapper stockFlowMapper;
    @Autowired
    private StockMapper stockMapper;

    @Scheduled(fixedDelay = 60000)  // 每分钟执行
    public void reconcile() {
        List<Long> skuIds = stockMapper.selectAllActiveSkuIds();
        for (Long skuId : skuIds) {
            try {
                reconcileSku(skuId);
            } catch (Exception e) {
                log.error("库存对账失败: skuId={}", skuId, e);
            }
        }
    }

    private void reconcileSku(Long skuId) {
        // 1. 获取Redis中的库存
        int redisStock = bucketStockService.getTotalStock(skuId);

        // 2. 从DB流水计算期望库存
        StockDO dbStock = stockMapper.selectBySkuId(skuId);
        int totalDeducted = stockFlowMapper.sumDeductedToday(skuId);
        int expectedRedisStock = dbStock.getInitStock() - totalDeducted;

        // 3. 比较差异
        int diff = redisStock - expectedRedisStock;
        if (Math.abs(diff) > 0) {
            log.warn("库存不一致: skuId={}, redis={}, expected={}, diff={}",
                skuId, redisStock, expectedRedisStock, diff);
            // 告警
            alertService.sendAlert("库存对账不一致", "skuId=" + skuId + ", diff=" + diff);
            // 自动修正（小差异直接修正，大差异人工介入）
            if (Math.abs(diff) <= 10) {
                bucketStockService.adjustStock(skuId, -diff);  // 修正
            }
        }
    }
}
```

### 12.2 Redis ZSet 实现

```
数据结构：
  key: "leaderboard:global"
  value: userId
  score: 用户积分

ZSet特性：
  - ZADD: O(logN)   更新分数
  - ZRANK: O(logN)  查询排名
  - ZRANGE: O(logN + M) 查询Top M
  - 自动按score排序
```

```java
@Service
public class LeaderboardService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private static final String GLOBAL_BOARD = "leaderboard:global";

    // 更新用户分数
    public void updateScore(Long userId, double scoreDelta) {
        redisTemplate.opsForZSet().incrementScore(GLOBAL_BOARD, userId.toString(), scoreDelta);
    }

    // 查询用户排名（1-based）
    public Long getUserRank(Long userId) {
        Long rank = redisTemplate.opsForZSet().reverseRank(GLOBAL_BOARD, userId.toString());
        return rank == null ? null : rank + 1;
    }

    // 查询用户分数
    public Double getUserScore(Long userId) {
        return redisTemplate.opsForZSet().score(GLOBAL_BOARD, userId.toString());
    }

    // 查询Top N榜单（附带分数）
    public List<RankItem> getTopN(int n) {
        Set<ZSetOperations.TypedTuple<Object>> items =
            redisTemplate.opsForZSet().reverseRangeWithScores(GLOBAL_BOARD, 0, n - 1);
        if (items == null) return Collections.emptyList();

        List<RankItem> result = new ArrayList<>();
        int rank = 1;
        for (ZSetOperations.TypedTuple<Object> item : items) {
            RankItem rankItem = new RankItem();
            rankItem.setRank(rank++);
            rankItem.setUserId(Long.valueOf(item.getValue().toString()));
            rankItem.setScore(item.getScore());
            result.add(rankItem);
        }
        return result;
    }

    // 查询某个用户周边排名（上下各N名）
    public List<RankItem> getNeighborRanks(Long userId, int neighborCount) {
        Long rank = redisTemplate.opsForZSet().reverseRank(GLOBAL_BOARD, userId.toString());
        if (rank == null) return Collections.emptyList();

        long start = Math.max(0, rank - neighborCount);
        long end = rank + neighborCount;

        Set<ZSetOperations.TypedTuple<Object>> items =
            redisTemplate.opsForZSet().reverseRangeWithScores(GLOBAL_BOARD, start, end);
        // 转换并返回...
        return convertWithRank(items, (int) start + 1);
    }
}
```

### 12.3 周榜/月榜实现

```java
@Service
public class TimedLeaderboardService {

    @Autowired
    private LeaderboardService leaderboardService;
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    // 获取当周排行榜Key
    private String getWeeklyKey() {
        // 格式: leaderboard:weekly:2024-W01
        LocalDate now = LocalDate.now();
        int weekOfYear = now.get(WeekFields.ISO.weekOfWeekBasedYear());
        int year = now.getYear();
        return String.format("leaderboard:weekly:%d-W%02d", year, weekOfYear);
    }

    // 获取当月排行榜Key
    private String getMonthlyKey() {
        LocalDate now = LocalDate.now();
        return String.format("leaderboard:monthly:%d-%02d", now.getYear(), now.getMonthValue());
    }

    // 同时更新全榜、周榜、月榜
    public void updateAllBoards(Long userId, double scoreDelta) {
        // 使用Pipeline批量执行，减少网络RTT
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            byte[] userIdBytes = userId.toString().getBytes();
            double delta = scoreDelta;

            connection.zIncrBy(GLOBAL_BOARD.getBytes(), delta, userIdBytes);
            connection.zIncrBy(getWeeklyKey().getBytes(), delta, userIdBytes);
            connection.zIncrBy(getMonthlyKey().getBytes(), delta, userIdBytes);
            return null;
        });

        // 设置周榜/月榜过期时间
        redisTemplate.expire(getWeeklyKey(), 14, TimeUnit.DAYS);  // 保留2周
        redisTemplate.expire(getMonthlyKey(), 40, TimeUnit.DAYS); // 保留40天
    }

    // 查询周榜Top 100
    public List<RankItem> getWeeklyTop100() {
        String key = getWeeklyKey();
        Set<ZSetOperations.TypedTuple<Object>> items =
            redisTemplate.opsForZSet().reverseRangeWithScores(key, 0, 99);
        return convertWithRank(items, 1);
    }
}
```

### 12.4 亿级用户分片方案

```
问题：ZSet单Key超过1亿成员时内存和性能下降

方案：分片ZSet

  userId % 100 -> 100个分片
  leaderboard:global:shard:0
  leaderboard:global:shard:1
  ...
  leaderboard:global:shard:99

查询Top N时：
  1. 从每个分片取Top N（共100*N条记录）
  2. 在应用层合并排序，取最终Top N

  注意：这是近似Top N，对于精确排名需要全量扫描
```

```java
@Service
public class ShardedLeaderboardService {

    private static final int SHARD_COUNT = 100;
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private String getShardKey(Long userId) {
        return "leaderboard:global:shard:" + (userId % SHARD_COUNT);
    }

    public void updateScore(Long userId, double scoreDelta) {
        String key = getShardKey(userId);
        redisTemplate.opsForZSet().incrementScore(key, userId.toString(), scoreDelta);
    }

    public Long getUserRank(Long userId) {
        // 1. 获取当前用户的分数
        String userKey = getShardKey(userId);
        Double userScore = redisTemplate.opsForZSet().score(userKey, userId.toString());
        if (userScore == null) return null;

        // 2. 统计所有分片中分数 > 当前用户的人数（并行查询优化）
        long higherCount = 0;
        for (int i = 0; i < SHARD_COUNT; i++) {
            String shardKey = "leaderboard:global:shard:" + i;
            // 统计score严格大于userScore的成员数
            Long count = redisTemplate.opsForZSet()
                .count(shardKey, userScore + 0.000001, Double.MAX_VALUE);
            if (count != null) higherCount += count;
        }
        return higherCount + 1;
    }

    public List<RankItem> getTopN(int n) {
        // 从每个分片取Top N
        List<RankItem> candidates = new ArrayList<>();
        for (int i = 0; i < SHARD_COUNT; i++) {
            String shardKey = "leaderboard:global:shard:" + i;
            Set<ZSetOperations.TypedTuple<Object>> items =
                redisTemplate.opsForZSet().reverseRangeWithScores(shardKey, 0, n - 1);
            if (items != null) {
                items.forEach(item -> {
                    RankItem ri = new RankItem();
                    ri.setUserId(Long.valueOf(item.getValue().toString()));
                    ri.setScore(item.getScore());
                    candidates.add(ri);
                });
            }
        }

        // 合并排序取Top N
        return candidates.stream()
            .sorted(Comparator.comparingDouble(RankItem::getScore).reversed())
            .limit(n)
            .map(item -> {
                item.setRank(candidates.indexOf(item) + 1);
                return item;
            })
            .collect(Collectors.toList());
    }
}
```

### 12.5 排行榜面试考点总结

```
1. 为什么用Redis ZSet而不是MySQL ORDER BY?
   - ZSet O(logN) vs MySQL全表扫描O(N)
   - Redis内存操作 vs MySQL磁盘I/O
   - ZSet自动维护有序性

2. 如何处理并发更新？
   - ZINCRBY是原子操作，天然并发安全

3. 亿级用户如何处理？
   - 分片ZSet（用户ID取模）
   - Top N查询：各分片取TopN，应用层合并
   - 精确排名：二分查找 + 分片并行查询

4. 如何保证实时性？
   - 每次得分变化直接ZINCRBY
   - 避免定时计算（延迟高）

5. 历史榜单如何保存？
   - 周榜/月榜设置独立Key
   - 定时任务结束时快照到MySQL
   - ZSet设置TTL，到期自动清理
```

### Q1: CAP理论在实际系统中的应用

```
CAP定理：一致性(C)、可用性(A)、分区容忍性(P)，最多同时满足两个

实际上P（网络分区）无法避免，所以选择CA还是CP：

CP系统（一致性优先）：
  - ZooKeeper、etcd
  - 适合：配置中心、分布式锁、Leader选举
  - 特点：网络分区时拒绝写入，保证不返回旧数据

AP系统（可用性优先）：
  - Cassandra、Eureka、DNS
  - 适合：用户个人设置、购物车、社交Feed
  - 特点：网络分区时仍可读写，但可能读到旧数据（最终一致性）

BASE理论（AP系统的实践）：
  - Basically Available: 基本可用
  - Soft state: 软状态（允许中间状态）
  - Eventually consistent: 最终一致性
```

### Q2: 如何设计一个高可用的MySQL架构

```
+------------------+
|    VIP (虚拟IP)   |
+------------------+
        |
+-------+-------+
|               |
[Master]  ->  [Slave1]  (半同步复制)
               [Slave2]
               [Slave3]

高可用组件：
  - MHA（Master High Availability）
  - Orchestrator
  - MySQL Group Replication

故障切换流程：
  1. 监控Master心跳
  2. 检测到Master宕机（多个节点确认）
  3. 选出最新的Slave（binlog最新）
  4. 提升为新Master
  5. 更新VIP指向新Master
  6. 其他Slave重新指向新Master
  总切换时间：通常30s~2min
```

### Q3: 分布式事务解决方案

```
方案1: 2PC（两阶段提交）
  问题：同步阻塞，协调者单点

方案2: TCC（Try-Confirm-Cancel）
  Try: 预留资源
  Confirm: 确认提交
  Cancel: 回滚释放
  适合：金融、库存等强一致性场景

方案3: Saga（推荐微服务）
  拆分为本地事务序列
  失败时执行补偿事务
  适合：电商下单、旅行预订

方案4: 消息最终一致性（推荐）
  本地事务 + 本地消息表
  定时扫描未发送消息
  消费方幂等处理
  适合：对一致性要求不极端的场景

推荐框架: Seata（支持AT/TCC/Saga/XA模式）
```

### B.2 MySQL 慢查询优化checklist

```
1. EXPLAIN分析执行计划
   - type: ALL(全表) -> range/ref/eq_ref(索引)
   - key: NULL -> 有索引名（确认索引被使用）
   - rows: 预估扫描行数（越小越好）
   - Extra: Using filesort / Using temporary（需要优化）

2. 索引优化原则
   - 最左前缀匹配（联合索引）
   - 区分度高的列放在索引左边
   - 避免在索引列上做函数操作（会失效）
   - 覆盖索引：SELECT的列都在索引中（无需回表）

3. 常见慢查询模式
   - SELECT *（返回不需要的列，无法覆盖索引）
   - LIKE '%xxx'（前缀模糊，索引失效）
   - OR条件（可能不走索引，改用UNION）
   - 大偏移量分页 LIMIT 100000,10（改用游标分页）
```

### B.3 Redis 性能优化

```
1. 禁止使用的命令（生产环境）

   SMEMBERS    -> 大Set用 SSCAN
   FLUSHALL/FLUSHDB -> 禁止
   DEBUG SLEEP -> 禁止

2. Pipeline批量操作
   // 一次网络RTT执行多个命令

       for (String key : keys) {

3. 连接池配置

   spring.redis.lettuce.pool.max-idle=20

4. 序列化优化
   // 使用Kryo/Protobuf替代Java序列化

```

### B.4 Kafka 最佳实践

```
生产者关键配置：
  acks=all                    # 所有副本确认（最高可靠性）
  retries=3                   # 失败重试次数
  batch.size=16384            # 批次大小（16KB）

消费幂等性设计：
  方案1: 数据库唯一约束（INSERT IGNORE）
```

### B.5 微服务治理

```
服务注册与发现：
  [Service A] -> 注册 -> [Nacos/Consul]
  [Service B] -> 发现 -> [Nacos/Consul]
  [Service B] -> 调用 -> [Service A实例列表] -> 负载均衡

负载均衡策略（Ribbon/LoadBalancer）：
  - RoundRobin: 轮询（默认）
  - Random: 随机
  - WeightedResponseTime: 响应时间加权
  - BestAvailable: 最小并发数

链路追踪（Sleuth + Zipkin）：
  每个请求生成traceId，跨服务透传
  通过Zipkin UI查看完整调用链
  找出高延迟的服务节点

配置中心（Nacos）：
  动态配置更新，无需重启服务
  灰度发布：按IP/用户比例推送配置
  配置版本管理，一键回滚
```
